# 端對端測試計畫：照片相簿整理應用程式

**功能分支**: `001-photo-albums`
**建立日期**: 2026-07-22

## US-1 建立相簿並整理照片（優先級：P1）

### Scenario: S-1-1 建立「旅行」相簿並一次匯入多格式照片

```gherkin
Scenario: 建立「旅行」相簿並一次匯入多格式照片
  Given 使用者尚未建立任何相簿
  And 本機有可選的 JPEG、PNG 與 WebP 照片
  When 使用者建立一個名為「旅行」的相簿並一次將這些照片加入「旅行」相簿
  Then 「旅行」相簿存在
  And 該相簿內含剛加入的那些照片
```

**對應欄位**:

- US-1 建立相簿並整理照片（優先級：P1）— 使用者可以建立多個相簿，將照片放入對應相簿中，讓原本零散的照片依主題或用途被整理起來。
- AC-1-1 假設使用者尚未建立任何相簿，當建立一個名為「旅行」的相簿並一次匯入多張 JPEG、PNG 與 WebP 照片，則系統應顯示「旅行」相簿且內含這些照片
- US1-FR1 系統必須允許使用者建立相簿並命名
- US1-FR2 系統必須允許使用者將照片加入指定相簿
- US1-FR3 系統必須支援一次選取多張 JPEG、PNG 或 WebP 照片匯入，並加入指定相簿

### Scenario: S-1-2 將已歸屬照片改加入另一相簿後只留在目標相簿

```gherkin
Scenario: 將已歸屬照片改加入另一相簿後只留在目標相簿
  Given 使用者已建立相簿 A 與相簿 B
  And 某張照片已屬於相簿 A
  When 使用者將該照片加入相簿 B
  Then 該照片只出現在相簿 B
  And 該照片不再出現在相簿 A
```

**對應欄位**:

- US-1 建立相簿並整理照片（優先級：P1）— 使用者可以建立多個相簿，將照片放入對應相簿中，讓原本零散的照片依主題或用途被整理起來。
- AC-1-2 假設使用者已建立多個相簿且某張照片已屬於相簿 A，當使用者將該照片加入相簿 B，則該照片應只出現在相簿 B，且不再出現在相簿 A
- Edge-G-2 當系統偵測到會造成同一張照片同時屬於多個相簿的操作時，必須改為將該照片移動到目標相簿，不得保留多重歸屬
- US1-FR4 當使用者將已屬於某相簿的照片加入另一相簿時，系統必須把該照片從原相簿移出，使其同一時間只屬於一個相簿


**對應欄位**:

- US-1 建立相簿並整理照片（優先級：P1）— 使用者可以建立多個相簿，將照片放入對應相簿中，讓原本零散的照片依主題或用途被整理起來。
- Edge-1-1 當使用者建立空相簿但尚未加入照片時，系統仍應顯示該相簿
- US1-FR1 系統必須允許使用者建立相簿並命名

### Scenario: S-1-4 選取不支援格式時拒絕匯入並說明支援範圍

```gherkin

  And 本機有至少一個非 JPEG、PNG、WebP 的檔案
  When 使用者嘗試將該不支援格式的檔案加入該相簿
  Then 系統拒絕匯入
  And 說明僅支援 JPEG、PNG、WebP
```

**對應欄位**:

- US-1 建立相簿並整理照片（優先級：P1）— 使用者可以建立多個相簿，將照片放入對應相簿中，讓原本零散的照片依主題或用途被整理起來。
- Edge-1-2 當使用者選取不支援的檔案格式時，系統應拒絕匯入並說明僅支援 JPEG、PNG、WebP
- US1-FR3 系統必須支援一次選取多張 JPEG、PNG 或 WebP 照片匯入，並加入指定相簿

---

### Scenario: S-1-5 拒絕相簿巢狀並維持單層

```gherkin
Scenario: 拒絕相簿巢狀並維持單層

  Then 系統阻止造成巢狀結構的行為
  And 所有相簿仍維持單層
```

**對應欄位**:

- Edge-G-1 當使用者嘗試以任何操作把相簿放進另一個相簿中時，系統必須阻止此行為

## US-2 在主頁面依日期瀏覽相簿（優先級：P2）

### Scenario: S-2-1 主頁依相簿建立日期分組顯示

```gherkin
Scenario: 主頁依相簿建立日期分組顯示
  Given 使用者已在不同日期建立多個相簿
  When 使用者進入主頁面
  Then 系統依各相簿的建立日期將相簿分組顯示
```

**對應欄位**:

- US-2 在主頁面依日期瀏覽相簿（優先級：P2）— 使用者回到主頁面時，可以看到所有相簿依建立日期分組顯示，藉此更快找到特定時間建立的照片集合。
- AC-2-1 假設使用者已在不同日期建立多個相簿，當使用者進入主頁面，則系統應依各相簿的建立日期將相簿分組顯示
- US2-FR1 系統必須在主頁面顯示所有相簿
- US2-FR2 系統必須依據相簿建立日期，在主頁面將相簿分組顯示

---

```gherkin
  Given 使用者已建立若干相簿
  And 某些日期下沒有任何相簿
  When 使用者進入主頁面
  Then 系統不顯示那些沒有相簿的空白日期分組
```

**對應欄位**:

- US-2 在主頁面依日期瀏覽相簿（優先級：P2）— 使用者回到主頁面時，可以看到所有相簿依建立日期分組顯示，藉此更快找到特定時間建立的照片集合。
- Edge-2-1 當某個日期分組下沒有任何相簿時，系統不應顯示空白日期分組
- US2-FR2 系統必須依據相簿建立日期，在主頁面將相簿分組顯示

## US-3 透過拖放重新排列相簿（優先級：P3）

### Scenario: S-3-1 拖放後主頁立即依新順序顯示相簿

```gherkin
Scenario: 拖放後主頁立即依新順序顯示相簿
  Given 主頁面上已有多個相簿
  When 使用者將其中一個相簿拖放到新的位置
  Then 系統依新的順序顯示相簿
```

**對應欄位**:

- US-3 透過拖放重新排列相簿（優先級：P3）— 使用者可以在主頁面直接拖放相簿，調整自己想要的排列順序，讓常用或重要的相簿排在更顯眼的位置。
- AC-3-1 假設主頁面上已有多個相簿，當使用者將其中一個相簿拖放到新的位置，則系統應依新的順序顯示相簿
- US3-FR1 使用者必須能夠在主頁面透過拖放重新排列相簿順序

---

### Scenario: S-3-2 拖放排序後重新整理或再進主頁仍保留順序

```gherkin
Scenario: 拖放排序後重新整理或再進主頁仍保留順序
  Given 使用者已在主頁完成一次相簿拖放排序
  When 使用者重新整理或再次進入主頁面
  Then 系統仍依先前保存的順序顯示相簿
```

**對應欄位**:

- US-3 透過拖放重新排列相簿（優先級：P3）— 使用者可以在主頁面直接拖放相簿，調整自己想要的排列順序，讓常用或重要的相簿排在更顯眼的位置。
- AC-3-2 假設使用者已完成一次相簿拖放排序，當使用者重新整理或再次進入主頁面，則系統應仍依先前保存的順序顯示相簿
- US3-FR2 系統必須保存使用者重新排列後的相簿順序

---

- Edge-3-1 當主頁面只有一個相簿時，拖放操作不應造成錯誤或混淆的互動結果

## US-4 在相簿內以平鋪方式預覽照片（優先級：P3）

### Scenario: S-4-1 打開含多張照片的相簿以平鋪預覽顯示

```gherkin
Scenario: 打開含多張照片的相簿以平鋪預覽顯示
  Given 某個相簿內已有多張照片
  When 使用者打開該相簿
  Then 系統以平鋪式預覽顯示相簿中的照片
```

**對應欄位**:

- US-4 在相簿內以平鋪方式預覽照片（優先級：P3）— 使用者進入任一相簿後，可以一次看到多張照片的縮圖平鋪排列，快速掃描相簿內容。
- AC-4-1 假設某個相簿內已有多張照片，當使用者打開該相簿，則系統應以平鋪式預覽顯示相簿中的照片
- US4-FR1 系統必須允許使用者進入單一相簿查看內容
- US4-FR2 系統必須在相簿內以平鋪式介面預覽照片

---

### Scenario: S-4-2 空相簿打開時顯示可理解的空狀態

```gherkin
Scenario: 空相簿打開時顯示可理解的空狀態
  Given 某個相簿內尚無照片
  When 使用者打開該相簿
  Then 系統顯示可理解的空狀態
  And 不是一片空白畫面
```

**對應欄位**:

- US-4 在相簿內以平鋪方式預覽照片（優先級：P3）— 使用者進入任一相簿後，可以一次看到多張照片的縮圖平鋪排列，快速掃描相簿內容。
- Edge-4-1 當相簿內尚無照片時，系統應顯示可理解的空狀態，而不是空白畫面
- US4-FR1 系統必須允許使用者進入單一相簿查看內容
- US4-FR3 系統必須在相簿沒有照片時顯示可理解的空狀態

## 假設

- 本計畫只依同 package 的 `spec.md` 展開；不讀取 `system-analyze/`，GWT 使用領域語言，不綁定具體 API 或頁面操作實作。
- Gherkin 慣例：`Given`／`Then` 可用 `And` 延續；`When` 僅一條操作步驟（不在 When 後接 And）。
- AC 與全域邊界語意重複時合併為單一 Scenario，並在「對應欄位」同時列出（見 S-1-2）。
- 本期無 `[NEED CLARIFICATION]` 邊界，故無 blocked Scenario。
- Edge ID（Edge-x-y／Edge-G-y）為本計畫為追溯穩定而編號；Edge-x-y 對應各 US「邊界情境」條列順序，Edge-G-y 對應「全域邊界情境」條列順序。
- 拖放與日期分組的共存細節以規格假設為準（主頁依建立日期分組；拖放調整排列順序並持久化）；本計畫不另綁定實作層的組內／跨組規則。
