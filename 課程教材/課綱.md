# 我們的專業分析｜州巧科技的現況

1. 目前不同角色對於需求釐清、系統分析、介面合約、測試設計與漸進式實作的理解深度並不一致，尚未形成可共同遵循的能力基線與開發流程。
2. 即使部分成員已理解 BDD（Behavior-Driven Development）/ TDD（Test-Driven Development）/ SDD（Specification-Driven Development） 的價值，後端團隊仍缺乏從需求釐清、系統分析、介面合約、測試計畫到漸進式實作的完整能力，因此規格即使寫成 Gherkin，在角色交接後容易出現理解落差、分析斷點與實作品質不一致等問題。

# 我們的客製化課程提議

## 我們點出的問題

1. 若在尚未建立基礎工程能力的情況下直接導入 BDD，團隊可能只多學會 Gherkin 或 Feature File 的形式，卻仍無法將規格可靠地轉化為 API、資料模型、測試與程式碼。要讓 BDD 真正成為 PM、SA、工程師的共同語言，必須先讓工程團隊具備承接規格、分析規格、驗證規格與實作規格的共同能力基線。
2. 如果把期望寄託在內訓課上，那我們就必須幫大家建立足夠的信心，避免內訓課程流於形式。那信心必定來自於「足夠的軟工基礎」，避免「見樹不見林」——「只知道 Gherkin 卻不敢改流程」、「只知道要寫測試，卻不知道測試從什麼上游元素推理而來」。

## 我們的專業提議

1. 我們建議將州巧科技的導入規劃為兩個階段。第一階段的客製化課程先以 【軟體工程實踐 × Artifact-First 規格驅動開發 × 測試驅動開發】 為主，並以後端為主要實戰範例。
2. 第一堂課我們將補齊團隊從需求到實作所需的基礎工程能力，讓成員不只會操作 AI 工具，而是能理解需求、系統分析、介面合約、E2E 測試計畫、TDD 與規格一致性之間的推理關係，並能依照貴公司的專案架構、API、Data 與開發流程進行客製化。
3. **如何銜接 BDD 課程？**
    1. 第一階段將先讓團隊實際體驗 AI × TDD 如何降低實作的不確定性，並建立對測試驅動開發的共同認知。當團隊開始認同：「好的，現在我們都覺得測試能保護實作 -- 對於產能和程式碼品質有幫助，那麼我們該如何減少測試程式碼的 Review 時間和意圖正確性？」之後，第二階段才能進一步討論：如何確保這些測試所表達的，正是 PM、SA 與使用者真正期待的行為，那這時就能推動 BDD。
    2. 因此，AI x BDD 第二堂內訓課，會直接切入重點，此時就不會再提到已經學過的 SDD 部分，而是直接講 BDD 的實踐流程，直接帶入工具實戰，從 DSL-Level Gherkin 開始建構認知，然後講技術 (Cucumber 生態系)，再提到 4-Rules Pattern，一樣先感受實戰效果，之後再拆解流程。
    3. AI x BDD 課程要學的就是：如何在整體系統分析流程中，加入 AI x BDD 的環節，此環節會進一步將 PM、SA、工程師 串接在共同規格之上，建立從需求範例、驗收規則、Feature Scenario 到自動化測試與 TDD 實作的完整協作流程，讓大家感受到說：「嗯，當大家願意在開發前共同投入 DSL-Level Gherkin 的釐清，後續的測試設計、TDD 實作、驗收與跨角色溝通，都能建立在同一份可執行規格之上。」藉此達到自然合作、學習的意願。

> 此兩階段設計，來自我們過往企業導入的實際經驗。
> 
> 
> 過去曾有企業希望直接採購單日 AI × BDD 內訓，但由於該團隊尚未建立共同的軟體工程與測試基礎，課後雖然大家能理解工具與形式，卻難以轉化為穩定的團隊實踐。其後，我們對此企業提議：使用優惠導入案的方式合作，來彌補內訓成效，才終於把 AI x BDD 直接導入到他們公司的專案中，成為團隊實踐。
> 
> 因此我們本次在課程設計上，更注重軟體工程實踐以及舉一反三的能力。
> 

# 第一階段課程：內訓成效

完成內訓後，團隊將能：

1. 熟悉軟體工程的基礎：看懂 SDD 與傳統 SDLC 的完整對應。
2. 熟悉軟體開發流程的分析：畫出 Artifact-First 軟體開發流程。
3. 掌握關鍵 SDD 專業：
    1. 將模糊需求轉化為可執行規格。
    2. 系統分析：從需求推導 API、Data、UI 等介面合約，並懂得增加新的系統分析環節、舉一反三。
    3. 快速審查系統分析產物的品質。
    4. 檢查多份規格之間的一致性與推理斷點。
4. 掌握可執行規格之專業：
    1. 設計能驗證需求的 E2E 測試計畫。
    2. 以 AI × TDD 控制開發節奏與實作品質。
5. 掌握 Skill 模組化的關鍵工程知識（SOP、Rules、Templates）：不只是程式碼要模組化，Skills 也要模組化，才能舉一反三。
6. 建立符合團隊架構與工程標準的客製化 AI 開發流程，無論是抽換規格 Artifact、調整系統分析顆粒度、擴充 TDD 流程、制定架構約束都知道怎麼做。

# 第一階段課程：課程議程

- 9:00 - 10:00 開場＆SDD 規格驅動開發（Specification-Driven Development）詳解
    - SDD 規格驅動開發（Specification-Driven Development）詳解、軟工概念流程介紹
    - 建立認知：什麼是 SDD 4+2 流程？傳統軟體開發 SDLC 和現代 SDD 流程之對應
    - 建立分析專業
        - Artifact-First 分析方法是什麼？怎麼畫圖？
        - SDD 4+2 流程，以主流工具 SpecKit 為例，畫出 Artifact-First 流程，看見軟體開發所需的規格推理。
- 10:00 - 12:00 【直接軟工實戰】Artifact-First SDD x TDD 5+2 開發流程上手實操
    - 下載我們特製的工具（Speckit 改良版，更適合因應團隊客製化需求的改良版本）
    - 直接使用實戰情境逐步操作完整 Artifact-First SDD x TDD 5+2 開發流程
        - AI x 需求初步撰寫：根據需求，訪談且拆解使用者需求、需求釐清後，產出規格（含用戶故事和驗收標準）
            - 關鍵軟工概念：
                - 需求分為哪幾種？FR（Functional Requirements）vs. NFR（Non-Functional Requirements）？
                - 需求的分層結構
                - 需求釐清有幾個維度？
        - AI x 系統分析流程：
            - 進行系統分析，盤點系統介面，產出相對應的介面合約，如： API、資料模型、UI 設計
            - 關鍵軟工概念：
                - 系統介面是什麼？需要幾個介面合約？
                - 每一個介面合約的標準怎麼選？主流標準有哪些？
                - 如何快速 Review 每個介面合約？
        - AI x 端對端測試計劃（以後端為例）：根據系統分析文件與規格，設計系統驗證（端對端）測試計劃
            - 關鍵軟工概念：
                - E2E Test 是什麼？
                - 測試計劃是什麼？測試計劃如何分層撰寫？
                - 什麼是好的測試計劃？什麼是壞的測試計劃？
                - 如何快速 Review 測試計劃？
        - AI x 實作計劃：根據 AI x TDD Skill Workflow（/tdd-e2e-red, /tdd-e2e-green, tdd-e2e-recfactor）產出系統實作計劃
            - 關鍵軟工概念：
                - 測試驅動開發 (TDD) 是什麼？為何必要？價值主張？
                - TDD 每一個環節的關鍵成功指標是什麼？如何 Review？
        - AI x 專案實作：根據系統實作計劃，逐步進行系統開發實作
        - AI x 需求釐清：確認流程內的產出物，是否有有模糊需要與使用者對齊的細節，進行訪談確認與規格補充
        - AI x 規格一致性分析：執行每個階段的產出物是否具備一至性確認
            - 關鍵軟工概念：
                - 使用 Artifact-First 的方式分析多份規格之間的推理關係，看見關鍵一致性
                - 如何 Review AI 的提議？權重是什麼？
- 12:00 - 13:00 午休
- 13:00 - 16:00 【客製化訓練】把 Artifact-First SDD x TDD 5+2 開發流程客製落地到團隊開發流程內
    - 看懂我們所改良後的模組化 Skills，不要只會用 skills 卻沒能力讀懂它，要看透 Skill 的模組化組成 (SOP, rules, templates)，並且敢改它
    - 實戰 -- 兩大客製化的方法：專案憲法 vs. 直接改 Skills
    - 實戰訓練：
        - 如何抽換系統分析產出的 Artifacts?
        - 如何加強系統分析的顆粒度，在某個分析環節加注更多方法論或是更細膩的推理？
        - 如何修改 TDD 時每一個步驟的執行方法？
        - 如何修改 TDD 的流程？讓後端能夠掛上資料庫設計的環節？
        - 如何修改架構約束？讓 AI 在生成程式碼時，必須配合架構約束？
        - 如何修改 API 和 Data 的設計約束？
        - 如何新增新的系統分析步驟？好讓團隊懂得舉一反三？
- 16:00 - 17:00 Q&A 時間
