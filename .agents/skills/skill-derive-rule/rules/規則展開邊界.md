# Rule 1 - 只有判準與限制才應抽成 RuleFile

- Level: `MUST`
- 應抽成 RuleFile 的內容，必須是用來把關 step 執行品質的判準、限制、禁止事項或常見失誤防呆。
- 若某段內容描述的是 phase 順序、必要動作、交付物產出或流程前後關係，應留在原 step 中，不應抽離成 RuleFile。
- 不可把原本用來告訴執行者「現在要做什麼」的流程骨架整段搬進 RuleFile。

## Good Example

- 這個例子是好的，因為它把「如何判斷是否該展開規則」抽成 rule，但把真正的流程動作留在原 step。

````md
## Phase 1 -- 收斂規則展開範圍

1. READ 讀取目標 SOP skill 內容，定位使用者指定的 phase 與 step。
2. THINK 若需要判斷哪些要求應抽成 RuleFile、哪些內容仍應留在原 step，先讀取 `rules/規則展開邊界.md`，再依該 step 的目的、輸入輸出與常見失誤收斂規則範圍。
````

## Bad Example

- 這個例子是壞的，因為它把原本的流程動作一起抽走，導致主 SOP 失去當下可執行性。

````md
## Phase 1 -- 收斂規則展開範圍

1. READ 讀取目標 SOP skill 內容。
2. THINK 直接把這一步的分析流程、後續委派方式與回寫方式都移到 `rules/規則展開邊界.md`，主 SOP 不再描述當下要做什麼。
````

# Rule 2 - 原 step 必須保留最小可執行動作

- Level: `MUST`
- 展開規則後，原 step 仍必須讓執行者知道當下要做的核心動作與目的。
- RuleFile 只補充判斷標準，不取代 step 本身的動詞、處理對象與階段目的。
- 若移除 RuleFile 後，step 會變成只剩抽象提醒或失去行動指向，表示展開邊界過度。

## Good Example

- 這個例子是好的，因為即使不展開 rule，step 仍然清楚要求先做判斷，再收斂規則範圍。

````md
2. THINK 若需要判斷哪些要求應抽成可按需載入的 RuleFile、哪些內容仍應留在原 step，先讀取 `rules/規則展開邊界.md`，再根據該 step 的目的、輸入輸出與常見失誤收斂規則邊界。
````

## Bad Example

- 這個例子是壞的，因為 step 只剩「去看規則」，沒有交代看完之後當下要完成的動作。

````md
2. THINK 若有需要，讀取 `rules/規則展開邊界.md`。
````

# Rule 3 - 優先從輸入、輸出與常見失誤收斂規則主題

- Level: `SHOULD`
- 在判斷是否值得抽成 RuleFile 時，應優先檢查該 step 的輸入、輸出與常見失誤是否存在穩定且可重複使用的把關標準。
- 若要求只屬於一次性的上下文說明，或不會反覆影響品質判斷，通常不值得抽成獨立 RuleFile。
- 同一份 RuleFile 應盡量圍繞單一判斷主題，避免把多種無關限制混寫在一起。

## Good Example

- 這個例子是好的，因為它把「輸入輸出與常見失誤的判斷邊界」整理成單一主題，適合獨立維護。

````md
# Rule 3 - 優先從輸入、輸出與常見失誤收斂規則主題

- Level: `SHOULD`
- 在判斷是否值得抽成 RuleFile 時，應優先檢查該 step 的輸入、輸出與常見失誤是否存在穩定且可重複使用的把關標準。
````

## Bad Example

- 這個例子是壞的，因為它把命名偏好、phase 切分、委派格式與步驟邊界全混在同一份 rule，主題過散。

````md
# Rule 4 - 展開時所有事情都一起規定

- Level: `SHOULD`
- 檔名怎麼取、phase 怎麼切、何時委派、Example 怎麼寫、什麼算常見失誤都寫在同一份規則裡。
````
