# Rule 1 - 預設執行方式必須寫成 `uv run <script.py>`

- Level: `MUST`
- 在 Skill 中提到執行 `scripts/` 下的 Python script 時，預設執行方式必須寫成 `uv run <script.py>`。
- 這個預設路徑應同時適用於有無第三方套件的腳本，避免執行者在 `python`、虛擬環境與手動安裝之間自行猜測。
- 不可把 `python <script.py>`、`./script.py` 或平台特定 launcher 當成主要執行方式。

## Good Example

- 這個例子是好的，因為 step 已直接指定統一且跨平台的執行命令。

```md
1. DELEGATE 若需要把輸入 JSON 正規化，執行 `uv run scripts/normalize_payload.py --input tmp/request.json`，再使用腳本輸出回填後續步驟。
```

## Bad Example

- 這個例子是壞的，因為它把執行方式留給執行者猜，也隱含依賴本機既有環境已手動裝好套件。

```md
1. DELEGATE 執行 `python scripts/normalize_payload.py`。
```

# Rule 2 - 缺少 `uv` 時必須提示標準 fallback，不可改用猜測式安裝

- Level: `MUST`
- 若執行環境缺少 `uv`，應優先提示安裝 `uv`，再重跑原本的 `uv run <script.py>`。
- 若暫時不能安裝 `uv`，且環境中的 `pip` 支援 `--requirements-from-script`，可先執行 `python -m pip install --requirements-from-script <script.py>`，再執行 `python <script.py>`。
- 不可根據 `ModuleNotFoundError` 臨時猜測套件名逐一安裝，也不可要求執行者自行閱讀 import 清單反推依賴。

## Good Example

- 這個例子是好的，因為它把依賴來源固定在 script 檔頭，並提供一致的備援路徑。

```md
若 `uv` 不可用，先提示安裝 `uv` 後執行 `uv run scripts/fetch_report.py`。
若當下只能使用 `pip 26+`，先執行 `python -m pip install --requirements-from-script scripts/fetch_report.py`，再執行 `python scripts/fetch_report.py`。
```

## Bad Example

- 這個例子是壞的，因為它讓執行者根據錯誤訊息猜套件，依賴處理方式不可預測也不可維護。

```md
若腳本失敗，就看缺哪個模組再自己 `pip install`。
```

# Rule 3 - 腳本與執行說明不可依賴特定作業系統的 shell 行為

- Level: `MUST`
- script 與 Skill step 不可把 `chmod +x`、Bash-only 語法、`.sh` 包裝器、POSIX 路徑假設或其他 OS 專屬 shell 行為當成必要前提。
- 若需要傳遞路徑、參數或輸出檔位置，應使用 Python 本身可攜的方式與明確參數，而不是倚賴某個 shell 展開規則。
- shebang 可以作為額外便利，但不可成為主要或唯一的執行方式。

## Good Example

- 這個例子是好的，因為主要路徑仍是跨平台的 `uv run`，沒有要求先做 POSIX 專屬設定。

```md
1. DELEGATE 執行 `uv run scripts/export_notes.py --source data/input.md --output data/output.json`。
```

## Bad Example

- 這個例子是壞的，因為它把 Linux 或 macOS 習慣當成所有平台都成立的前提。

```md
1. DELEGATE 先執行 `chmod +x scripts/export_notes.py`，再用 `./scripts/export_notes.py` 跑腳本。
```
