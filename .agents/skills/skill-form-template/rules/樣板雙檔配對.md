# Rule 1 - 每份樣板都必須同時提供骨架檔與範例檔

- Level: `MUST`
- 每份樣板都必須以同 basename、同副檔名的兩個檔案成對存在：骨架檔使用 `<name>.<ext>`，範例檔使用 `<name>.example.<ext>`。
- 兩個檔案都必須放在目標 skill 的 `templates/` 目錄下。
- 不可只建立骨架檔而缺少範例檔，也不可只保留範例檔而沒有骨架檔。

## Good Example

- 這個例子是好的，因為它提供了同主題、同副檔名的完整雙檔配對。

```text
templates/class-diagram.mmd
templates/class-diagram.example.mmd
```

## Bad Example

- 這個例子是壞的，因為它缺少對應範例檔，執行者無法同時看到骨架與完整成品。

```text
templates/class-diagram.mmd
```

# Rule 2 - 骨架檔與範例檔的責任必須清楚分開

- Level: `MUST`
- 骨架檔只放固定結構、固定語法與待改寫的填位符號，讓執行者可以直接複製後再填值。
- 對應範例檔必須提供同結構的具體成品，讓執行者知道填完後應長成什麼樣子。
- 不可讓骨架檔塞滿一次性的具體內容，也不可讓範例檔只是再貼一份尚未替換的填位符號。

## Good Example

- 這個例子是好的，因為骨架檔保留結構與填位符號，範例檔則提供對照用的完整內容。

```md
骨架檔 `templates/class-diagram.mmd`
classDiagram
class {{CLASS_NAME}} {
  +{{FIELD_NAME}}: {{FIELD_TYPE}}
}

範例檔 `templates/class-diagram.example.mmd`
classDiagram
class Order {
  +id: UUID
}
```

## Bad Example

- 這個例子是壞的，因為骨架檔與範例檔都沒有扮演清楚角色，無法指導複製與改寫。

```md
骨架檔 `templates/class-diagram.mmd`
classDiagram
class Order {
  +id: UUID
}

範例檔 `templates/class-diagram.example.mmd`
classDiagram
class {{CLASS_NAME}} {
  +{{FIELD_NAME}}: {{FIELD_TYPE}}
}
```

# Rule 3 - 樣板名稱應直指產物主題與格式

- Level: `SHOULD`
- 樣板名稱應讓人一眼看出它要生成的產物主題，而不是只用抽象或暫時性名稱。
- 副檔名應與實際輸出格式一致，避免讓骨架檔與最終產物格式脫節。
- 同一組雙檔應只對應單一產物主題，避免把多種無關格式混進同一個樣板名稱下。

## Good Example

- 這個例子是好的，因為檔名與副檔名都直接對齊要生成的 Mermaid 類別圖。

```text
templates/class-diagram.mmd
templates/class-diagram.example.mmd
```

## Bad Example

- 這個例子是壞的，因為檔名過度抽象，無法快速判斷這組樣板要生成什麼。

```text
templates/template.txt
templates/template.example.txt
```
