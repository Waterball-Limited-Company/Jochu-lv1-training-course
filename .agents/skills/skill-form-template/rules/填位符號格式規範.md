# Rule 1 - 填位符號必須使用 `{{UPPER_SNAKE_CASE}}` 格式

- Level: `MUST`
- 骨架檔中的填位符號必須使用雙大括號包住大寫蛇形命名，例如 `{{CLASS_NAME}}`、`{{FIELD_TYPE}}`。
- 填位符號名稱必須表達語意，不可使用 `{{VALUE1}}`、`{{TEXT}}` 這類過度抽象的命名。
- 若同一段內容需要多個填位符號，應讓每個名稱都能對應到明確待填欄位。

## Good Example

- 這個例子是好的，因為填位符號的包法一致，且名稱直接對應待填內容。

```md
class {{CLASS_NAME}} {
  +{{FIELD_NAME}}: {{FIELD_TYPE}}
}
```

## Bad Example

- 這個例子是壞的，因為填位符號格式不一致，且名稱也無法清楚表意。

```md
class [name] {
  +{{value1}}: <<type>>
}
```

# Rule 2 - 同一語意的填位符號應保持一致命名

- Level: `SHOULD`
- 同一份骨架檔中，若多處代表同一語意槽位，應重複使用同一個填位符號名稱。
- 不應把同一語意拆成多個不同名稱，否則執行者難以判斷哪些位置應同步改寫。
- 只有在語意真的不同時，才應拆成不同的填位符號。

## Good Example

- 這個例子是好的，因為同一個類別名稱在多處都使用 `{{CLASS_NAME}}`，改寫時不會混淆。

```md
class {{CLASS_NAME}} {
  +{{FIELD_NAME}}: {{FIELD_TYPE}}
}
{{CLASS_NAME}} --> {{RELATED_CLASS_NAME}}
```

## Bad Example

- 這個例子是壞的，因為同一語意被拆成多種命名，容易讓人誤以為要填不同值。

```md
class {{CLASS_NAME}} {
  +{{FIELD_NAME}}: {{FIELD_TYPE}}
}
{{ENTITY_NAME}} --> {{RELATED_CLASS_NAME}}
```

# Rule 3 - 填位符號只覆蓋可變內容，不覆蓋固定格式符號

- Level: `MUST`
- 填位符號只應標示真正會變動的內容區段，固定語法、分隔符號、標題符號與結構關鍵字應保留在骨架中。
- 不可把 Markdown 標記、程式語法或圖表語法整段包進填位符號，否則骨架會失去格式引導作用。
- 若某段文字同時包含固定格式與可變內容，應只讓填位符號覆蓋可變部分。

## Good Example

- 這個例子是好的，因為 Markdown 標題與 Mermaid 語法都保留，只把實際會替換的值抽成填位符號。

```md
## {{SECTION_TITLE}}

class {{CLASS_NAME}} {
  +{{FIELD_NAME}}: {{FIELD_TYPE}}
}
```

## Bad Example

- 這個例子是壞的，因為它把固定格式一起包進填位符號，骨架不再提供穩定結構。

```md
{{## SECTION_TITLE}}

{{class CLASS_NAME {
  +FIELD_NAME: FIELD_TYPE
}} 
```
