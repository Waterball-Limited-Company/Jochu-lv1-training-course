# Rule 1 - 最終 e2e-test-plan 必須寫到固定路徑

- Level: `MUST`
- `e2e-test-plan` 的最終產物路徑固定為 `specs/<NNN-plan-package>/e2e-test-plan.md`。
- 產物與同 package 的 `spec.md` 並列，不放在 `system-analyze/` 底下。
- 不可把完成稿只留在聊天訊息、暫存筆記或其他任意檔名中。

## Good Example

- 這個例子是好的，因為輸出路徑與 skill 契約完全對齊。

```text
specs/001-photo-albums/e2e-test-plan.md
```

## Bad Example

- 這個例子是壞的，因為把端對端測試計畫塞進系統分析目錄，與上游契約混淆。

```text
specs/001-photo-albums/system-analyze/e2e-test-plan.md
```

# Rule 2 - `plan-package` 必須與上游 spec 目錄對齊

- Level: `MUST`
- `plan-package` 必須沿用同功能的 `specs/<NNN-plan-package>/` 目錄名，採 `NNN-` 三位數前綴加英文小寫 kebab-case slug。
- 若使用者已指定既有 package，應寫入該 package，不可另開平行目錄。
- 若尚無 `spec.md`，應先請使用者完成 `/specify` 或指定既有 package，不可自行發明 package 名稱。

## Good Example

- 這個例子是好的，因為 e2e-test-plan 與 spec 落在同一 package。

```text
specs/003-reading-list/spec.md
specs/003-reading-list/e2e-test-plan.md
```

## Bad Example

- 這個例子是壞的，因為測試計畫與 spec 分散在不同 package，追溯會斷開。

```text
specs/003-reading-list/spec.md
specs/004-e2e-only/e2e-test-plan.md
```

# Rule 3 - 標題與 metadata 必須對齊 spec 樣式

- Level: `MUST`
- 檔案主標題固定為 `# 端對端測試計畫：{{FEATURE_NAME}}`。
- 標題下方必須分行列出：`**功能分支**:`、`**建立日期**:`、`**狀態**:`，格式對齊同 package 的 `spec.md`。
- `**功能分支**:` 的值必須與所在 `plan-package` 相同（可用 code span）。

## Good Example

- 這個例子是好的，因為標題與 metadata 與 spec 可互相追溯。

```md
# 端對端測試計畫：照片相簿整理

**功能分支**: `001-photo-albums`
**建立日期**: 2026-07-20
**狀態**: 草稿
```

## Bad Example

- 這個例子是壞的，因為把 metadata 擠成一行且欄位名稱與 spec 不一致。

```md
# E2E Test Plan: 照片相簿整理

**功能**: 001-photo-albums ｜ **日期**: 2026-07-20 ｜ **來源**: spec.md
```
