# Rule 1 - 每則 Scenario 必須有「對應欄位」且只列有對到的項目

- Level: `MUST`
- 每則 Scenario 在 Gherkin 區塊之後必須有 `**對應欄位**:` 標題。
- 條列只能寫有實際對應的項目，格式為 `- <ID> <說明>`；ID 可為 `US-*`、`AC-*`、`Edge-*`、`FR-*`。
- 不可使用「User Story」「驗收標準」「邊界條件」「FR」等分類小標；沒有對到的類型就省略，不寫「無對應」佔位句。

## Good Example

- 這個例子是好的，因為扁平條列且只含有對到的資料。

```md
**對應欄位**:

- US-1 手動建立相簿…（優先級：P1）— …
- AC-1-1 假設…當…則…
- FR-001 系統必須允許使用者建立新的相簿…
```

## Bad Example

- 這個例子是壞的，因為用分類標題包一層，且硬寫無對應 AC。

```md
**對應**:

- **User Story**: US-1 …
- **驗收標準**: （本 Scenario 無對應 AC）
- **FR**:
  - FR-002 …
```

# Rule 2 - 對應欄位的說明必須接上 spec 原文語意，不可只留編號

- Level: `MUST`
- 每條對應列在 ID 之後必須接上足以對齊 `spec.md` 的說明文字（US 標題與敘事摘要、AC／Edge／FR 的完整或等價表述）。
- 不可只寫 `- US-1`、`- FR-001` 這類裸編號。
- 建議順序：先 US，再 AC 與／或 Edge，最後 FR；同一類型可多列。

## Good Example

- 這個例子是好的，因為編號後直接帶 spec 說明。

```md
- FR-003 系統必須確保相簿只能以單層結構存在，不可把相簿放進另一個相簿
```

## Bad Example

- 這個例子是壞的，因為只有編號無法 Review 對齊。

```md
- FR-003
```

# Rule 3 - 檔末必須有元素覆蓋總表，欄位為 ID／描述／Scenario

- Level: `MUST`
- 檔案必須包含覆蓋總表（可用「元素覆蓋總表」或等價標題），採一列一個被追蹤元素。
- 表格欄位固定為 `ID | 描述 | Scenario`，不可再加「類型」欄。
- 每個 US、每條 AC、每條 Edge（含 blocked）、每條 FR 都必須出現；已覆蓋者填 Scenario 編號，未澄清者不產 Scenario 者填 `blocked`。
- 同一 Scenario 可出現在多個元素列；矩陣與各 Scenario 對應欄位必須一致。

## Good Example

- 這個例子是好的，因為元素覆蓋可直接掃出缺口。

```md
| ID | 描述 | Scenario |
| --- | --- | --- |
| AC-1-1 | … | S-1-1 |
| Edge-1-3 | … | blocked |
| FR-001 | … | S-1-1 |
```

## Bad Example

- 這個例子是壞的，因為用類型欄且缺少元素列粒度。

```md
| 類型 | ID | Scenario |
| --- | --- | --- |
| Scenario | S-1-1 | US-1, AC-1-1, FR-001 |
```
