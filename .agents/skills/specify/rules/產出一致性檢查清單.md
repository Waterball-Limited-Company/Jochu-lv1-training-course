# Rule 1 - 輸出前必須檢查所有輸入語意是否都有落點

- Level: `MUST`
- 在交付最終 spec 前，必須逐一檢查每個語意片段是否已被映射到 User Story、Global Rule、待澄清項、關鍵實體或假設。
- 若有任何高影響語意片段沒有落點，應先修補，不得直接輸出。

## Good Example

- 這個例子是好的，因為它逐項核對了 prompt 的五個核心語意。

```md
- 整理照片到相簿 -> US1
- 依日期分組 -> US2
- 拖放重排 -> US3
- 平鋪預覽 -> US4
- 不可嵌套 -> GR-001
```

## Bad Example

- 這個例子是壞的，因為它遺漏了全域限制的落點。

```md
- 整理照片到相簿 -> US1
- 依日期分組 -> US2
- 拖放重排 -> US3
- 平鋪預覽 -> US4
```

# Rule 2 - 必須檢查故事內結構是否完整且編號連續

- Level: `MUST`
- 每個 User Story 都必須同時具備：描述、優先級理由、獨立驗證方式、功能需求、成功標準、驗收情境與邊界情境。
- `USn-FRm` 與 `USn-SCm` 編號必須和對應故事編號一致，且在故事內連續遞增。
- 若存在 `GR-xxx`，其編號也應連續遞增。

## Good Example

- 這個例子是好的，因為故事內結構完整且編號一致。

```md
### 使用者故事 4 - 在相簿內以平鋪方式預覽照片

**功能需求**
- **US4-FR1**：...
- **US4-FR2**：...

**成功標準**
- **US4-SC1**：...
```

## Bad Example

- 這個例子是壞的，因為故事編號與需求編號不一致，且欄位缺漏。

```md
### 使用者故事 4 - 在相簿內以平鋪方式預覽照片

**功能需求**
- **US3-FR1**：...
```

# Rule 3 - 全域約束、待澄清項與 placeholder 位置都必須正確

- Level: `SHOULD`
- 交付前應確認全域限制沒有誤放到故事內，待澄清項沒有濫用，且最終內容不殘留 `{{PLACEHOLDER}}`。
- 若輸出要作為 benchmark 等級成品，應再用 validator 做一次靜態檢查。

## Good Example

- 這個例子是好的，因為它把內容放在正確位置，且沒有殘留骨架內容。

```md
## 共通規則與全域約束

- **GR-001**：所有相簿都必須位於同一層級，且不可嵌套在其他相簿之中
```

## Bad Example

- 這個例子是壞的，因為它保留了 placeholder，且把待澄清項寫得太抽象。

```md
- **US2-FR2**：系統必須依據 [NEEDS CLARIFICATION: 還沒決定] 來進行日期分組
- **US4-SC1**：{{USER_STORY_SC_1}}
```
