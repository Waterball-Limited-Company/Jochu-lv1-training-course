# Rule 1 - 先把原始敘述切成 spec 的輸入單位

- Level: `MUST`
- 在開始切故事前，必須先把使用者輸入切成 `初次輸入`、補充輸入、主要角色、外部互動對象、明確範圍、非範圍與全域限制。
- 只有先切出這些單位，後續才不會把補充條件誤當新故事，或把非目標誤寫成 FR。
- 若同一段敘述同時包含範圍與限制，應拆開放到各自適合的區塊，而不是整段照抄到某一處。

## Good Example

- 這個例子是好的，因為它先把原始需求拆成後續可編排的 spec 單位。

```md
- 初次輸入：我想做一個線上課程平台。
- 補充輸入 1：第一版先做註冊登入、講師建課、送審、上架、購買、修課、作業、退款。
- 補充輸入 2：先不要做直播課、討論區、團購、訂閱制會員、多語系。
- 主要角色：訪客、學員、講師、營運人員。
- 外部互動對象：金流系統。
```

## Bad Example

- 這個例子是壞的，因為它沒有先切出範圍與限制，後續很容易把所有資訊混成同一層。

```md
- 需求：做一個線上課程平台，反正上面提到的都做。
```

# Rule 2 - 只有使用者可觀察的價值才往下切成故事

- Level: `MUST`
- 能獨立被使用者或營運角色感知、操作、驗證的價值，才適合切成 User Story。
- API 風格、輪詢、時區、金額格式、附件型別等支援性限制，預設先留在 `原始需求` 或 `假設`，不要直接切成獨立故事。
- 若某條限制會直接改變某個故事的驗收方式，應保留它在全域區，並只在受影響故事內反映其效果。

## Good Example

- 這個例子是好的，因為它把使用者價值與支援性限制分開放置。

```md
User Story：
- 學員購買付費課程

全域限制：
- 前後端走 RESTful API
- 付款等待頁以輪詢更新狀態
```

## Bad Example

- 這個例子是壞的，因為它把實作偏好誤切成和使用者價值同層的故事。

```md
### 使用者故事 7 - 前後端改用 RESTful API
### 使用者故事 8 - 付款狀態用輪詢更新
```

# Rule 3 - 非目標與低確定性資訊應顯式保留，而不是被腦補成承諾

- Level: `SHOULD`
- 使用者明確排除的內容應保留為非目標或限制，不應在故事內重新長成功能承諾。
- 使用者只給方向、尚未拍板的內容，應先留在補充輸入、假設或待澄清標記，而不是直接補完成具體 FR。
- 若某段敘述對第一版只提供上下文價值，應保留原意，不為了「看起來完整」而過度具體化。

## Good Example

- 這個例子是好的，因為它保留了範圍邊界，沒有擅自擴張承諾。

```md
## 假設

- 第一版不處理直播課、討論區、團購、訂閱制會員、多語系
```

## Bad Example

- 這個例子是壞的，因為它把排除項目反向腦補成後續一定要做的規格。

```md
- **FR-099**: 系統必須支援未來的直播課、討論區與團購功能
```
