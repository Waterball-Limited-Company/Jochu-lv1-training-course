# Rule 1 - 會改變 spec 主結構的缺口必須先澄清

- Level: `MUST`
- 若缺口會改變 User Story 邊界、角色權限、公開性規則、狀態轉換、交易規則、或驗收單位，就必須先向使用者澄清。
- 在這類情況下，不可直接用直覺補完，否則後續故事切分、FR 歸屬與驗收標準都可能失真。
- 若使用者當下可互動，應優先呼叫 `/clarify` 收斂，而不是把所有高影響缺口都延後到 spec 內。

## Good Example

- 這個例子是好的，因為它先問會改變故事邊界的高影響決策。

```md
先澄清：
- 未驗證 Email 的使用者能否登入？
- 退款成功後是否要立即失去修課資格？
```

## Bad Example

- 這個例子是壞的，因為它直接腦補了會影響故事與驗收的核心規則。

```md
未驗證 Email 的使用者一律不能登入。
退款成功後保留 30 天修課權限。
```

# Rule 2 - 無法即時拍板的局部缺口要用 `[NEED CLARIFICATION: ...]` 顯式標記

- Level: `MUST`
- 若某個缺口重要但尚未拿到答案，且它只局部影響某個故事或邊界條件，應在最受影響的位置用 `[NEED CLARIFICATION: ...]` 顯式標記。
- 不可把這類缺口默默省略，也不可假裝它已被決定。
- 標記內容必須是一句具體問題，讓後續讀者知道還缺哪個決策。

## Good Example

- 這個例子是好的，因為它把未決策缺口留在受影響故事內，方便之後回頭補。

```md
- 當文章已被下架時，是否允許保留已儲存項目仍是 [NEED CLARIFICATION: 下架文章是否保留在閱讀清單中?]
```

## Bad Example

- 這個例子是壞的，因為它讓讀者無法分辨是需求沒有提，還是產出者忘記整理。

```md
- 當文章已被下架時，系統行為待定
```

# Rule 3 - 低風險預設可帶著假設推進，但必須保持可回頭修正

- Level: `SHOULD`
- 若某項資訊不會改變故事切分，只影響措辭、預設呈現方式或低風險操作細節，可先以 `## 假設` 顯式記錄後繼續。
- 這類假設應保持最小承諾，不應因為缺少資訊就發展成大量細節。
- 一旦使用者後續補充與假設衝突，應優先回寫 spec，而不是捍衛原先假設。

## Good Example

- 這個例子是好的，因為它把低風險預設留在全域假設區，沒有污染故事主體。

```md
## 假設

- 清單的預設呈現方式沿用產品既有的列表閱讀習慣
```

## Bad Example

- 這個例子是壞的，因為它把低風險預設膨脹成多條高承諾 FR。

```md
- **FR-021**: 閱讀清單必須固定採三欄卡片式瀑布流版型
- **FR-022**: 每個卡片必須帶陰影與收藏動畫
```
