# Rule 1 - 每個使用者故事只承載一個可獨立感知的使用者價值

- Level: `MUST`
- 每個 User Story 都必須對應一個可單獨描述、單獨驗證的使用者價值。
- 若兩個能力讓使用者感知到不同的結果，即使它們同屬同一畫面，也應拆成不同的故事。
- 不可因為它們出現在同一句 prompt 中，就把多個價值硬併成單一故事。

## Good Example

- 這個例子是好的，因為它把瀏覽、拖放與預覽拆成不同故事。

```md
- 使用者故事 2：在主頁面依日期瀏覽相簿
- 使用者故事 3：透過拖放重新排列相簿
- 使用者故事 4：在相簿內以平鋪方式預覽照片
```

## Bad Example

- 這個例子是壞的，因為它把三種不同價值混成同一故事。

```md
- 使用者故事 2：在主頁面依日期瀏覽、拖放重排並平鋪預覽相簿內容
```

# Rule 2 - 只有缺乏獨立使用者價值的必要子步驟才應合併到同一故事

- Level: `MUST`
- 若某個能力只是另一個核心故事的必要子步驟，且單獨不構成獨立價值，應合併到同一故事。
- 合併判斷應以使用者完成任務時感知到的整體成果為準，而不是以操作步驟多少為準。
- 不可把單純前後相依的能力一律拆開，造成故事破碎。

## Good Example

- 這個例子是好的，因為建立相簿與將照片放入相簿共同形成「整理照片」的核心價值。

```md
- 使用者故事 1：建立相簿並整理照片
  - 建立相簿
  - 將照片加入指定相簿
```

## Bad Example

- 這個例子是壞的，因為它把同一個核心整理任務拆得過碎。

```md
- 使用者故事 1：建立相簿
- 使用者故事 2：把照片加入相簿
```

# Rule 3 - 切分故事時應優先維持故事數量精簡但完整

- Level: `SHOULD`
- 在資訊足夠時，應優先產出最少且完整的故事集合，避免故事數量過多導致層級失焦。
- 若同一份需求可合理落在 3 到 6 個故事內，優先維持這個範圍。
- 只有當新增故事能明顯提升獨立驗證性或階層清楚度時，才應再拆新故事。

## Good Example

- 這個例子是好的，因為它用四個故事完整覆蓋 benchmark 的主要價值。

```md
- US1：建立相簿並整理照片
- US2：在主頁面依日期瀏覽相簿
- US3：透過拖放重新排列相簿
- US4：在相簿內以平鋪方式預覽照片
```

## Bad Example

- 這個例子是壞的，因為它把同一條主要價值拆成過多微型故事。

```md
- US1：建立相簿
- US2：命名相簿
- US3：打開主頁
- US4：查看日期
- US5：拖放
- US6：查看縮圖
```
