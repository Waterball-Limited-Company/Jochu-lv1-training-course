# Rule 1 - 每個使用者故事都必須有可衡量的成功標準

- Level: `MUST`
- 每個 User Story 都必須有至少一條可衡量的成功標準，且成功標準必須掛在該故事內。
- 成功標準應描述使用者是否順利完成該故事的目標，或該故事是否達到可接受的體驗品質。
- 不可把所有成功標準集中到故事外的全域章節。

## Good Example

- 這個例子是好的，因為成功標準直接對應故事價值。

```md
### 使用者故事 3 - 透過拖放重新排列相簿

**成功標準**

- **US3-SC1**：90% 的使用者能在不閱讀說明的情況下，成功完成一次相簿拖放排序
```

## Bad Example

- 這個例子是壞的，因為成功標準沒有隸屬故事。

```md
## 成功標準

- **SC-001**：90% 的使用者能完成一次相簿拖放排序
```

# Rule 2 - 成功標準必須依故事類型套用固定量化模板

- Level: `MUST`
- 核心任務型故事，優先使用「`80%` 的使用者能在 `3 分鐘` 內完成第一次核心任務」類型模板。
- 瀏覽定位型故事，優先使用「使用者能在 `10 秒` 內找到目標內容」類型模板。
- 直接操作型故事，優先使用「`90%` 的使用者能在不閱讀說明下完成操作」類型模板。
- 內容呈現型故事，優先使用「使用者能在 `5 秒` 內看見所需內容」類型模板。
- 若 prompt 未提供更強的數據依據，應優先採用上述預設值，不可任意漂移數字。

## Good Example

- 這個例子是好的，因為它依故事類型套用穩定模板。

```md
- **US1-SC1**：80% 的使用者能在第一次使用時，於 3 分鐘內建立至少 1 個相簿並加入照片
- **US2-SC1**：使用者進入主頁面後，能在 10 秒內從正確日期分組找到目標相簿
- **US4-SC1**：使用者進入相簿後，能在 5 秒內看見該相簿內所有照片的平鋪式預覽
```

## Bad Example

- 這個例子是壞的，因為數值完全隨意，且沒有對應故事類型。

```md
- **US1-SC1**：53% 的使用者能在 17 分鐘內建立相簿
- **US2-SC1**：使用者大概很快就能找到相簿
- **US4-SC1**：照片應該要看起來舒服
```

# Rule 3 - 可用性痛點的負向回饋指標只在有明確風險時補充

- Level: `SHOULD`
- 若某個故事容易產生明確的可用性痛點，可再補一條負向回饋占比指標，例如「相關回饋低於 `10%`」。
- 負向回饋指標應補充故事的主要風險，而不是每個故事都機械性新增。

## Good Example

- 這個例子是好的，因為它只在操作與整理風險明顯的故事補充回饋指標。

```md
- **US1-SC2**：與「找不到照片整理位置」相關的回饋，在測試期間低於 10%
- **US3-SC2**：與「相簿順序難以調整」相關的回饋，在測試期間低於 10%
```

## Bad Example

- 這個例子是壞的，因為它把所有故事都硬套相同的回饋指標。

```md
- **US2-SC2**：與「日期分組顯示」相關的回饋，在測試期間低於 10%
- **US4-SC2**：與「平鋪式預覽」相關的回饋，在測試期間低於 10%
```
