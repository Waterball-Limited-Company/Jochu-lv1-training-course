# Rule 1 - 優先級必須依使用者價值依賴關係排序

- Level: `MUST`
- `P1` 應保留給沒有它就無法成立產品核心價值的故事。
- `P2` 應保留給在核心物件已存在後，使用者最主要的瀏覽、定位或理解能力。
- `P3` 應保留給效率提升、個人化操作或內容呈現增強等非阻斷性價值。

## Good Example

- 這個例子是好的，因為它讓優先級反映使用者任務依賴關係。

```md
- P1：建立相簿並整理照片
- P2：在主頁面依日期瀏覽相簿
- P3：透過拖放重新排列相簿
- P3：在相簿內以平鋪方式預覽照片
```

## Bad Example

- 這個例子是壞的，因為它把視覺呈現擺到核心任務之前。

```md
- P1：在相簿內以平鋪方式預覽照片
- P2：建立相簿並整理照片
```

# Rule 2 - 當多個故事都不是核心建立能力時，先排主要導航，再排優化操作與呈現

- Level: `MUST`
- 若多個故事都依賴核心故事成立，應先排序會影響主要導航與定位的故事，再排序提升效率的操作故事與呈現故事。
- 對同樣屬於非核心故事的候選，若其中一個直接影響使用者找到目標內容，應優先於純互動優化。
- 不可只依句子順序決定 `P2` 與 `P3`。

## Good Example

- 這個例子是好的，因為日期瀏覽先於拖放與平鋪預覽。

```md
- P2：在主頁面依日期瀏覽相簿
- P3：透過拖放重新排列相簿
- P3：在相簿內以平鋪方式預覽照片
```

## Bad Example

- 這個例子是壞的，因為它只照 prompt 描述順序排序。

```md
- P2：透過拖放重新排列相簿
- P3：在主頁面依日期瀏覽相簿
```

# Rule 3 - 優先級理由必須說明前後依賴，而不是重述故事內容

- Level: `SHOULD`
- 每個故事的優先級理由應說明它對前後故事的依賴關係、核心性或風險，而不是單純把故事內容換句話說。
- 若故事被排成較低優先級，理由中應明確點出其前提依賴。

## Good Example

- 這個例子是好的，因為理由說明了拖放排序建立在已能建立與瀏覽相簿之上。

```md
拖放排序能提升瀏覽效率與個人化體驗，但前提是使用者已能建立相簿並在主頁面看到它們，因此順位較低。
```

## Bad Example

- 這個例子是壞的，因為它只是重複故事敘述，沒有說明排序依據。

```md
因為使用者可以拖放排序，所以這是一個拖放排序的故事。
```
