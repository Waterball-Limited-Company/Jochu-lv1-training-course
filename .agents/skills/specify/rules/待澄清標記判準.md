# Rule 1 - 只有方向明確但關鍵參數缺失時，才可標記待澄清

- Level: `MUST`
- `[NEEDS CLARIFICATION: ...]` 只用於需求方向已明確，但仍缺少會影響實作或驗證的重要參數時。
- 若需求連方向都不清楚，應先進入 `/clarify`，而不是直接在 spec 內大量標記待澄清。
- 不可把模型單純不想決定的內容一律標成待澄清。

## Good Example

- 這個例子是好的，因為需求方向已經存在，只缺少決策軸細節。

```md
- **US2-FR2**：系統必須依據 [NEEDS CLARIFICATION: 需確認是依相簿建立日期、照片拍攝日期，或照片匯入日期] 來進行日期分組
```

## Bad Example

- 這個例子是壞的，因為整體需求方向都還沒收斂。

```md
- **US1-FR1**：系統必須 [NEEDS CLARIFICATION: 這個產品到底要做什麼]
```

# Rule 2 - 待澄清標記必須明說缺的是哪個決策軸

- Level: `MUST`
- 每個待澄清標記都必須清楚指出缺的是哪個決策軸，例如資料來源、排序依據、支援格式、權限邊界或觸發條件。
- 待澄清文字應讓使用者看完就知道要拍板什麼，不可只寫成抽象提醒。

## Good Example

- 這個例子是好的，因為它明白指出缺的是匯入方式與格式範圍。

```md
- **US1-FR3**：系統必須支援 [NEEDS CLARIFICATION: 需確認是否允許一次上傳多張照片，以及支援哪些照片格式] 的照片匯入方式
```

## Bad Example

- 這個例子是壞的，因為讀者看不出到底要決定什麼。

```md
- **US1-FR3**：系統必須支援 [NEEDS CLARIFICATION: 還有些細節要再想想] 的照片匯入方式
```

# Rule 3 - 待澄清項目數量應維持精簡，只保留高影響缺口

- Level: `SHOULD`
- 待澄清項目應只保留會影響故事邊界、功能需求、驗收方式或資料模型的高影響缺口。
- 若只是措辭偏好或低風險預設，應直接採用合理預設，不必新增待澄清標記。
- 同一故事內若有多個缺口，應優先合併成最少但清楚的待澄清條目。

## Good Example

- 這個例子是好的，因為它只保留 benchmark 中兩個高影響缺口。

```md
- 日期分組依據未明
- 照片匯入方式與格式範圍未明
```

## Bad Example

- 這個例子是壞的，因為它把低價值細節全部標成待澄清。

```md
- 相簿標題字數待澄清
- 空狀態文案待澄清
- 平鋪縮圖邊角圓角大小待澄清
```
