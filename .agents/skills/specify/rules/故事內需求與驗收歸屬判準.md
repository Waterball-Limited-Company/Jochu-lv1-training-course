# Rule 1 - FR 必須歸屬在對應的 User Story 內

- Level: `MUST`
- 每條 FR 都必須掛在某一個具體 User Story 底下，不可散落在故事外的全域需求區。
- FR 的存在是為了補強故事的可交付內容邊界，因此必須與該故事的敘事、優先序與驗證方式一起閱讀。
- 若某條需求找不到明確歸屬故事，表示故事切分可能有問題，應先回頭調整故事，而不是硬塞進全域區。

## Good Example

- 這個例子是好的，因為 FR 與故事價值保持同一層。

```md
### 使用者故事 1 - 儲存文章到閱讀清單 (優先級：P1)

- **FR-001**: 系統必須允許已登入的使用者從文章頁把文章加入個人閱讀清單
- **FR-002**: 系統必須避免同一篇文章在同一位使用者的閱讀清單中重複出現
```

## Bad Example

- 這個例子是壞的，因為 FR 脫離了故事，讀者無法判斷需求在驗證時屬於哪個單位。

```md
## 功能需求

- **FR-001**: 系統必須允許已登入的使用者從文章頁把文章加入個人閱讀清單

### 使用者故事 1 - 儲存文章到閱讀清單
```

# Rule 2 - 驗收標準、邊界條件與獨立驗證都要留在故事內

- Level: `MUST`
- 每個 User Story 都必須在自身區塊內包含 `如何獨立驗證此使用者故事`、`邊界條件` 與 `驗收標準`。
- 驗收標準應以外部可觀察結果為主，優先使用「假設 / 當 / 則」句型，避免退化成實作筆記。
- 邊界條件應描述故事在例外情境下的行為，不應改寫成和主流程重複的 FR。

## Good Example

- 這個例子是好的，因為同一個故事的驗收與邊界資訊都留在原位。

```md
**如何獨立驗證此使用者故事**: 在文章頁執行一次儲存操作，確認文章進入個人清單且不會因重複操作產生重複項目。

**邊界條件**:

- 當使用者重複儲存同一篇文章時，系統必須阻止重複新增並提供清楚回應

**驗收標準**:

1. **假設** 使用者已登入且正在閱讀一篇尚未儲存的文章，**當** 點擊儲存，**則** 該文章會被加入個人閱讀清單且顯示成功訊息
```

## Bad Example

- 這個例子是壞的，因為驗收與邊界條件被放到故事外，失去對應關係。

```md
### 使用者故事 1 - 儲存文章到閱讀清單

## 驗收標準

1. ...
```

# Rule 3 - 局部未決策應放在最受影響的故事內，而不是全域模糊帶過

- Level: `SHOULD`
- 若某個未決策只影響特定故事的邊界行為，應在該故事的 `邊界條件` 或 `驗收標準` 中標記 `[NEED CLARIFICATION: ...]`。
- 不應把局部未決策搬到全域區，導致讀者不知道它實際影響哪個故事。
- 只有真正跨故事的缺口，才適合放到全域假設或先回到澄清流程。

## Good Example

- 這個例子是好的，因為下架文章的處理方式只影響「查看閱讀清單」與「儲存文章」的邊界。

```md
**邊界條件**:

- 當文章已被下架時，是否允許保留已儲存項目仍是 [NEED CLARIFICATION: 下架文章是否保留在閱讀清單中?]
```

## Bad Example

- 這個例子是壞的，因為它把局部缺口抽離故事，讀者無法知道哪裡需要回頭修。

```md
## 假設

- 還有一些下架文章相關問題之後再說
```
