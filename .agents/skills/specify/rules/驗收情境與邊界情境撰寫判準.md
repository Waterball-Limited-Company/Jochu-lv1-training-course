# Rule 1 - 每個使用者故事都必須至少有一個直接對應核心需求的驗收情境

- Level: `MUST`
- 每個 User Story 至少要有一個驗收情境，且該情境必須直接驗證此故事最核心的 `FR`。
- 驗收情境應以實際可觀察的操作與結果撰寫，不可只重述需求標題。
- 若故事包含多個高重要度 `FR`，可增加多個情境，但仍應由同一個使用者價值串起。

## Good Example

- 這個例子是好的，因為情境直接對應「建立相簿並加入照片」。

```md
1. **假設** 使用者尚未建立任何相簿，**當** 使用者建立一個名為「旅行」的相簿並加入 5 張照片，**則** 系統應顯示「旅行」相簿且內含這 5 張照片
```

## Bad Example

- 這個例子是壞的，因為它只是把故事名換句話說，沒有可驗證操作。

```md
1. **假設** 使用者想整理照片，**當** 使用者整理照片，**則** 系統應該支援整理照片
```

# Rule 2 - 驗收情境必須使用固定的「假設 / 當 / 則」格式

- Level: `MUST`
- 每個驗收情境都必須使用「`假設` / `當` / `則`」格式，讓條件、動作與結果邊界清楚。
- 不可改成自由敘事段落，也不可缺少三段中的任一段。

## Good Example

- 這個例子是好的，因為條件、操作與結果都被明確拆出。

```md
1. **假設** 主頁面上已有多個相簿，**當** 使用者將其中一個相簿拖放到新的位置，**則** 系統應依新的順序顯示相簿
```

## Bad Example

- 這個例子是壞的，因為它沒有固定格式，條件與結果混在一起。

```md
1. 使用者拖放相簿之後，畫面應更新，且應該保留新的順序。
```

# Rule 3 - 邊界情境只保留故事局部例外，全域限制另寫在全域邊界情境

- Level: `SHOULD`
- 故事的 `邊界情境` 應只保留與該故事直接相關的空狀態、單一資料量、例外輸入或局部互動風險。
- 若某個例外是在驗證全域約束是否被守住，應移到 `全域邊界情境`，避免故事邊界情境混入全域規則。

## Good Example

- 這個例子是好的，因為它處理的是故事局部空狀態。

```md
**邊界情境**

- 當相簿內尚無照片時，系統應顯示可理解的空狀態，而不是空白畫面
```

## Bad Example

- 這個例子是壞的，因為它把全域巢狀限制塞進故事局部邊界。

```md
**邊界情境**

- 當使用者嘗試把相簿放進另一個相簿中時，系統必須阻止此行為
```
