# Rule 1 - 每條功能需求都必須歸屬到且只歸屬到一個使用者故事

- Level: `MUST`
- 每條 `FR` 都必須掛在單一 User Story 內，讓需求單位與驗證範圍清楚可追溯。
- `FR` 的歸屬應以該需求主要支持哪個使用者價值為準，而不是以它出現在哪個句子或畫面為準。
- 不可保留獨立的全域 `功能需求` 區塊，把故事內需求再次散落到故事外。

## Good Example

- 這個例子是好的，因為每條 `FR` 都明確掛在對應故事下。

```md
### 使用者故事 2 - 在主頁面依日期瀏覽相簿

**功能需求**

- **US2-FR1**：系統必須在主頁面顯示所有相簿
- **US2-FR2**：系統必須依據 [NEEDS CLARIFICATION: ...] 來進行日期分組
```

## Bad Example

- 這個例子是壞的，因為 `FR` 沒有明確隸屬故事。

```md
## 需求

- **FR-001**：系統必須在主頁面顯示所有相簿
- **FR-002**：系統必須依日期將相簿分組
```

# Rule 2 - 只有跨故事且永遠成立的限制才可寫成全域約束

- Level: `MUST`
- `共通規則與全域約束` 只用來放跨多個故事、對整個系統永遠成立的限制。
- 若某條規則主要只是支撐單一故事，應回到該故事的 `FR` 或邊界情境，不可因為看起來抽象就提升為全域。
- 全域約束應用 `GR-xxx` 編號，並與故事內 `FR` 明確分開。

## Good Example

- 這個例子是好的，因為「相簿不可嵌套」是跨故事且持續成立的結構性限制。

```md
## 共通規則與全域約束

- **GR-001**：所有相簿都必須位於同一層級，且不可嵌套在其他相簿之中
```

## Bad Example

- 這個例子是壞的，因為它把故事專屬需求錯放到全域。

```md
## 共通規則與全域約束

- **GR-001**：使用者必須能夠在主頁面拖放重新排列相簿
```

# Rule 3 - 全域約束的例外案例應留在全域邊界情境，不回流到單一故事

- Level: `SHOULD`
- 若某個例外案例是在驗證全域限制是否被遵守，應寫在 `全域邊界情境`，而不是重複散落到各故事的邊界情境。
- 單一故事的 `邊界情境` 應只保留與該故事直接相關的例外與空狀態。

## Good Example

- 這個例子是好的，因為它把違反全域限制的案例留在全域區。

```md
### 全域邊界情境

- 當使用者嘗試以任何操作把相簿放進另一個相簿中時，系統必須阻止此行為
```

## Bad Example

- 這個例子是壞的，因為它把同一個全域限制重複寫進多個故事。

```md
### 使用者故事 1
**邊界情境**
- 當使用者嘗試把相簿放進另一個相簿中時，系統必須阻止此行為

### 使用者故事 2
**邊界情境**
- 當使用者嘗試把相簿放進另一個相簿中時，系統必須阻止此行為
```
