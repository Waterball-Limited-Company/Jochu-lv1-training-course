# Rule 1 - 輸入需求必須先切成最小語意片段

- Level: `MUST`
- 在開始撰寫 spec 前，必須先把原始 prompt 切成最小且可獨立判讀的語意片段。
- 單一片段只承擔一個主要語意，例如核心任務、瀏覽方式、操作方式、呈現方式、限制或待澄清參數。
- 若一句話同時包含多個語意，必須拆開，不可整句直接對應單一 User Story。

## Good Example

- 這個例子是好的，因為它把同一句需求拆成多個可分流的片段。

```md
原始需求：
相簿按日期分組，並可以在主頁面上透過拖放來重新排列。

語意片段：
- 相簿按日期分組
- 使用者可以在主頁面拖放重新排列相簿
```

## Bad Example

- 這個例子是壞的，因為它把兩個不同的使用者價值綁成同一片段。

```md
原始需求：
相簿按日期分組，並可以在主頁面上透過拖放來重新排列。

語意片段：
- 使用者可以在主頁面依日期分組並拖放重新排列相簿
```

# Rule 2 - 每個語意片段都必須先分類再進入後續編排

- Level: `MUST`
- 每個語意片段都必須先分類為固定集合中的一類，再決定後續歸屬。
- 建議使用的分類為：`CORE_USER_GOAL`、`NAVIGATION_OR_DISCOVERY`、`DIRECT_MANIPULATION`、`VIEW_OR_PRESENTATION`、`GLOBAL_CONSTRAINT`、`NEEDS_CLARIFICATION`。
- 未經分類的片段不得直接被寫入 User Story、Global Rule 或成功標準。

## Good Example

- 這個例子是好的，因為每個片段都有明確類型，後續可穩定分流。

```md
- 幫助我將照片整理到不同的相簿中 -> CORE_USER_GOAL
- 相簿按日期分組 -> NAVIGATION_OR_DISCOVERY
- 相簿絕不會嵌套在其他相簿中 -> GLOBAL_CONSTRAINT
```

## Bad Example

- 這個例子是壞的，因為它直接跳到故事命名，沒有先完成分類。

```md
- 幫助我將照片整理到不同的相簿中 -> 使用者故事 1
- 相簿按日期分組 -> 使用者故事 2
- 相簿絕不會嵌套在其他相簿中 -> 先看看寫哪裡
```

# Rule 3 - 正規化時不得提前發明未被需求支持的方案

- Level: `SHOULD`
- 正規化輸入時應保留使用者已明示的需求方向，不應提前補入未被需求支持的技術方案、裝置限制或商業規則。
- 若缺的是關鍵參數而不是方向，應保留為待澄清項，而不是在切分階段自行補完。
- 同義詞可正規化成一致用詞，但不得改變原始需求邊界。

## Good Example

- 這個例子是好的，因為它保留需求方向，並把缺口留到待澄清。

```md
- 照片匯入方式未明 -> NEEDS_CLARIFICATION
- 日期分組依據未明 -> NEEDS_CLARIFICATION
```

## Bad Example

- 這個例子是壞的，因為它在沒有依據時提前補了產品決策。

```md
- 照片一定會從手機相簿匯入，且只支援 JPG
- 日期分組一律依照片拍攝日期
```
