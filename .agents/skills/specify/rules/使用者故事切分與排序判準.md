# Rule 1 - 使用者故事必須對應可獨立驗證的使用者價值

- Level: `MUST`
- 切分 User Story 時，必須以某個角色可感知、可操作、可獨立驗證的價值作為單位。
- 不可只因為資料表、頁面、欄位或 CRUD 動作不同，就把它們切成不同故事。
- 若多個步驟共同構成同一個核心價值，應優先收斂成同一故事，再把細節展開成該故事的 FR。

## Good Example

- 這個例子是好的，因為故事單位對應的是完整價值，而不是零碎操作。

```md
### 使用者故事 1 - 儲存文章到閱讀清單 (優先級：P1)

已登入的使用者在閱讀文章時，想先把文章存起來，之後再回來閱讀而不會遺失。
```

## Bad Example

- 這個例子是壞的，因為它只是把單一價值拆成多個低層操作。

```md
### 使用者故事 1 - 點擊儲存按鈕
### 使用者故事 2 - 寫入資料庫
### 使用者故事 3 - 顯示成功訊息
```

# Rule 2 - 優先級排序應反映價值鏈與前置依賴

- Level: `MUST`
- P1、P2、P3 的排序應優先反映核心價值、風險控制與前置依賴，而不是單純依使用者提到的順序抄寫。
- 每個故事都必須說明「為何是這個優先序」與「如何獨立驗證此使用者故事」，讓排序與驗證單位能互相對齊。
- 若某故事建立在另一故事之上，排序時應把依賴較少、核心價值較大的故事放前面。

## Good Example

- 這個例子是好的，因為它先交付核心價值，再處理建立在其上的延伸管理能力。

```md
P1：儲存文章到閱讀清單
P2：查看自己的閱讀清單
P3：移除不再需要的已儲存文章
```

## Bad Example

- 這個例子是壞的，因為它先排了依賴後續成果的故事，導致驗證順序混亂。

```md
P1：移除已儲存文章
P2：查看閱讀清單
P3：儲存文章到閱讀清單
```

# Rule 3 - 只有共享同一驗證單位的內容才應合併成同一故事

- Level: `SHOULD`
- 若兩段需求共享同一個角色目標、驗證方式與完成價值，應合併成同一故事，避免把 spec 切得過碎。
- 若兩段需求需要不同驗收方式、不同角色或不同前置條件，則應分開成不同故事。
- 合併與拆分的判斷應優先看驗證單位，而不是篇幅長短。

## Good Example

- 這個例子是好的，因為「建立草稿」與「編輯草稿」共同支撐講師完成建課這個價值。

```md
### 使用者故事 3 - 講師建立並維護課程草稿
```

## Bad Example

- 這個例子是壞的，因為它把同一個驗證單位拆成多個微型故事。

```md
### 使用者故事 3 - 輸入課程名稱
### 使用者故事 4 - 上傳封面圖
### 使用者故事 5 - 新增章節
```
