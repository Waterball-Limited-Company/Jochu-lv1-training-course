# Rule 1 - 跨故事共通資訊應集中放在全域區，不重複散落成 FR

- Level: `MUST`
- 角色共通權限、明確非目標、全域前提、格式限制、時間與金額規則、系統整合原則等跨故事資訊，應集中整理在 `# 原始需求` 或 `## 假設`。
- 這類資訊不可在多個故事中重複複製成 FR，否則容易造成矛盾或維護成本上升。
- 若某項全域規則會影響特定故事，只應在該故事內反映其驗收效果，而不是重新定義一次規則。

## Good Example

- 這個例子是好的，因為全域限制與故事內效果被清楚分層。

```md
## 假設

- 所有金額都用新台幣整數
- 所有時間都以平台時區為準

### 使用者故事 5 - 學員購買付費課程

- **FR-011**: 折扣套用後的訂單總金額不得為負數
```

## Bad Example

- 這個例子是壞的，因為它把同一條全域限制重複貼到多個故事裡。

```md
### 使用者故事 5 - 學員購買付費課程
- **FR-011**: 所有金額都用新台幣整數

### 使用者故事 8 - 學員申請退款
- **FR-031**: 所有金額都用新台幣整數
```

# Rule 2 - 真正跨故事的狀態與限制應整理成共通語彙，再回掛到故事效果

- Level: `SHOULD`
- 當使用者明確列出課程、訂單、修課資格、作業提交或退款申請等狀態時，應先在全域區整理成共通語彙，再讓各故事只承接與自己有關的狀態變化。
- 不應在每個故事裡各自發明新的狀態名稱或不同說法。
- 若狀態清單仍未定稿，應在最相關故事內標記待澄清，而不是一邊寫全域清單、一邊又在故事裡用別的名詞。

## Good Example

- 這個例子是好的，因為全域狀態與故事驗收之間保持同一套名詞。

```md
## 原始需求

- 訂單狀態：待付款、已付款、已取消、已退款

### 使用者故事 5 - 學員購買付費課程

1. **假設** 訂單仍在付款期限內，**當** 學員付款失敗後再次付款，**則** 原訂單仍維持待付款直到付款成功或逾時取消
```

## Bad Example

- 這個例子是壞的，因為全域與故事各自使用不同狀態名詞。

```md
## 原始需求

- 訂單狀態：待付款、已付款、已取消、已退款

### 使用者故事 5 - 學員購買付費課程

1. **假設** 交易處於 processing，**當** 付款失敗，**則** 狀態改為 expired
```
