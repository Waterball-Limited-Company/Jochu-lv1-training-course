# Rule 1 - 優化既有 skill 時必須先重演問題，再談改法

- Level: `MUST`
- 當使用者要優化既有 skill 時，必須先從不滿意的結果往回推，實際檢視目標 skill 或相關 skill chain 的 `SKILL.md`、相關 `rules/`、`templates/`、`scripts/`，再收斂問題來源。
- 不可只根據使用者一句抱怨、單一症狀描述或主觀猜測，直接提出新的編排方案或開始改檔。
- 若問題的形成過程牽涉到多個 skill 的上下游 handoff，應把重演範圍擴到相關 skill chain，而不是只盯著單一 skill 的單一檔案。

## Good Example

- 這個例子是好的，因為它先回推並重演現況，再根據實際觀察收斂問題，而不是直接猜修法。

````md
## Phase 2 -- 優化時完成根因確認閘門

1. THINK 若本次是優化既有 skill，先讀取 `rules/優化時先做根因分析與確認閘門.md`，再根據使用者描述的不滿意結果，回推並實際演練目標 skill 或 skill chain 的 SOP、相關 `rules/`、`templates/`、`scripts/`，收斂預期結果、現況、落差分析、根因、建議改動層級與候選刪除項。
````

## Bad Example

- 這個例子是壞的，因為它把「使用者不滿意」直接等同於「已經知道要怎麼改」。

````md
## Phase 2 -- 直接開始改

1. THINK 使用者說結果不好，所以直接新增更多 rules 與 templates。
````

# Rule 2 - 根因確認閘門必須交代預期、現況、落差、根因、改動層級與候選刪除項

- Level: `MUST`
- 優化既有 skill 的分析結果，必須至少交代使用者真正預期的結果、實際重演出的現況、兩者之間的落差分析、收斂出的根因、建議改動層級，以及候選刪除項。
- 不可只寫「我覺得哪裡怪怪的」或「我建議這樣改」，卻沒有先說清楚為什麼會失敗，以及為什麼該用這個改動層級。
- 候選刪除項可以是 phase、step、rule、template、script，重點是讓後續編排不被既有檔案拓樸綁住。

## Good Example

- 這個例子是好的，因為它讓後續提案建立在完整的落差分析與根因上。

````md
## 根因
- 根因類型: `Control-Plane Gap`
- 改動範圍: `skill rewrite`
- 失敗原因: 既有流程缺少 optimize lane 的根因確認閘門，導致所有優化任務都被導向加法式編排。

## 候選刪除項
- 舊版直接從問題跳到編排提案的 phase
- 舊版沒有 delete candidates 欄位的提案格式
````

## Bad Example

- 這個例子是壞的，因為它直接跳到修法，沒有先交代預期、現況、落差與根因。

````md
## 根因
- 建議直接新增一份 rule，應該就會比較好。
````

# Rule 3 - 問題涉及上下游 handoff 時，分析範圍必須擴到 skill chain

- Level: `MUST`
- 若問題來自某個 skill 的輸出被另一個 skill 誤用、漏接或放大，根因分析範圍必須擴到該 skill chain。
- 不可在 handoff 問題明顯存在時，硬把責任壓成單一 skill 的局部瑕疵。
- 只有在確認問題真的侷限在單一 skill 內部時，才適合把改動層級收斂成 local patch 或 step rewrite。

## Good Example

- 這個例子是好的，因為它把分析邊界擴到造成結果失真的上下游 skill。

````md
使用者反映最終輸出的 skill 缺少 template 掛接說明。

分析範圍：
- 讀取目標 skill
- 讀取上游負責產生 SOP 的 skill
- 讀取下游負責 derive template 的 skill

收斂結果：
- 問題不是單一 RuleFile，而是上游 step 沒有正確保留 template 掛接時機。
````

## Bad Example

- 這個例子是壞的，因為明明是上下游 handoff 問題，卻只在單一 skill 裡加註解了事。

````md
使用者反映最終輸出缺少 template 掛接說明。

處理方式：
- 只修改其中一個 template 檔的說明文字，不檢查上游或下游 skill。
````

# Rule 4 - 未取得根因確認閘門確認前，不得進入編排提案或檔案修改

- Level: `MUST`
- 完成根因確認閘門後，必須先向使用者說明觀點並取得確認，之後才能進入後續編排提案或檔案修改。
- 不可一邊提交根因分析，一邊同步提出實作方案並直接開始改檔。
- 這個確認閘門的目的，是先確認大家對問題與根因的理解一致，再決定後續如何重建控制平面與模組邊界。

## Good Example

- 這個例子是好的，因為它明確把根因確認閘門當成後續編排前的停點。

````md
1. WRITE 若本次是優化既有 skill，先讀取 `templates/skill-root-cause-report.md` 與 `templates/skill-root-cause-report.example.md`，再向使用者提交根因分析。
2. READ 若本次是優化既有 skill，讀取使用者對根因確認閘門的確認或修正意見；未收到確認前停止，不進入後續編排提案或檔案修改。
````

## Bad Example

- 這個例子是壞的，因為它把根因分析與後續修改混在同一個動作中，沒有真正形成閘門。

````md
1. WRITE 先寫根因分析，順便把 SKILL.md 一起改完。
````
