# Rule 1 - 不論是 create lane 還是 optimize lane，都必須先建立或重建 SOP 控制平面

- Level: `MUST`
- 在創建新 skill 或優化既有 skill 時，都必須先以最小可執行 SOP 作為控制平面，再決定後續模組化方式。
- 若既有 skill 的主流程已經過胖、失焦或 gate 錯位，應先重建 SOP，而不是直接在原有 step 外層加新規則或新樣板。
- 不可把 `rules/`、`templates/`、`scripts/` 的增修當成主要工程動作，卻放任主 SOP 繼續失真。

## Good Example

- 這個例子是好的，因為它先把主流程收斂成控制平面，再展開後續模組決策。

````md
## Phase 3 -- 重建控制平面與編排提案

1. DELEGATE 呼叫 `/skill-form-sop` 建立新 skill 的 SOP section，或把既有 skill 的 SOP 重建成最小可執行控制平面，讓後續編排都以同一版 SOP 為主幹。
````

## Bad Example

- 這個例子是壞的，因為它直接跳到新增模組，卻沒有先處理已失真的主流程。

````md
## Phase 3 -- 直接補模組

1. DELEGATE 先新增三份 rule 與兩份 template。
2. THINK 之後再看看要不要調整 SOP。
````

# Rule 2 - 編排時必須逐 step 做 keep-inline、rewrite、derive 或 delete 決策

- Level: `MUST`
- 進行 skill engineering 時，必須逐 step 判斷是保留 inline、重寫 step、展開 `rule`、展開 `template`、展開 `script`，或直接刪除。
- 不可把整份 skill 視為只能「整體保留」或「整體重寫」的二選一；真正的編排工作應落在 step 粒度。
- 也不可預設所有優化都只能做加法式 derivation，而不允許 delete 或 rewrite。

## Good Example

- 這個例子是好的，因為它把 step 決策顯式化，而不是籠統說「順便優化一下」。

````md
2. THINK 若需要判斷各 step 應保留 inline、重寫、展開 `rule`、展開 `template`、展開 `script`，或直接刪除，先讀取相關規則，再逐 step 收斂 keep-inline、rewrite、derive-rule、derive-template、derive-script、delete 的決策。
````

## Bad Example

- 這個例子是壞的，因為它沒有實際做 step 粒度判斷，只是抽象地說要模組化。

````md
2. THINK 讓整份 skill 變得更模組化。
````

# Rule 3 - 高判斷工作留在主 SOP，穩定判準、格式與自動化才抽成模組

- Level: `SHOULD`
- 若某個 step 的核心價值在需求理解、預期管理、設計取捨或 root cause 判讀，應優先保留在主 SOP，而不是外包成 `script` 或 `template`。
- 若某個 step 需要穩定的品質判準，適合抽成 `rule`；若需要穩定輸出骨架，適合抽成 `template`；若需要可重複執行的機械處理，適合抽成 `script`。
- 抽模組的目的，是降低主流程噪音並提高重用性，而不是把所有內容切碎。

## Good Example

- 這個例子是好的，因為它把高判斷工作留在 SOP，把可重複資源抽到對應模組。

````md
- Root Cause Gate 的預期對齊與 gap analysis 留在主 SOP。
- `rules/優化時先做根因分析與確認閘門.md` 補充分析邊界與 gate 規則。
- `templates/skill-root-cause-report.md` 提供固定輸出骨架。
- `scripts/analyze_skill_references.py` 負責機械式引用盤點。
````

## Bad Example

- 這個例子是壞的，因為它把高判斷工作錯誤地外包給不適合的模組。

````md
- 用 template 取代 root cause 判讀本身。
- 用 script 直接決定 skill 應該採用哪種改動層級。
````

# Rule 4 - 需要模組化落地時，應調度既有 deriver，而不是繞過生態系硬寫

- Level: `MUST`
- 當某個 step 已判斷需要 `rule`、`template`、`script` 模組時，應優先調度 `/skill-derive-rule`、`/skill-derive-template`、`/skill-derive-script` 完成落地。
- 不可先在主 SOP 中主張要維持模組化編排，實作時卻繞過既有 deriver 直接散寫各種模組內容。
- 這個要求是為了讓 skill engineering 自己也遵守同一套 skill 演化規則，而不是只要求別的 skill 遵守。

## Good Example

- 這個例子是好的，因為它把編排決策和模組落地交給對應 deriver 處理。

````md
1. DELEGATE 若某個 step 需要額外把關規則，呼叫 `/skill-derive-rule` 展開或重寫對應 `rules/` 模組，並把讀取時機回掛到主 SOP。
2. DELEGATE 若某個 step 需要穩定輸出骨架，呼叫 `/skill-derive-template` 展開或重寫對應 `templates/` 模組，並把讀取時機回掛到主 SOP。
3. DELEGATE 若某個 step 需要可重複自動化處理，呼叫 `/skill-derive-script` 展開或重寫對應 `scripts/` 模組，並把執行時機回掛到主 SOP。
````

## Bad Example

- 這個例子是壞的，因為它口頭上說要模組化，實際上卻直接越過既有演化機制。

````md
1. WRITE 直接手寫所有新的 RuleFile、template 與 script，之後再補回 SOP。
````

# Rule 5 - 主 SOP 只保留控制平面，不重述已委派資源內容

- Level: `MUST`
- 若某個 step 已明確要求讀取 `rule`、`template`、`script` 或委派其他 skill，主 SOP 應只保留該資源的讀取或執行時機、處理對象與預期產出。
- 不可在同一個 step 內重述該資源已承擔的細部判準、固定欄位、範例內容或長段補充說明，否則會破壞單一溯源。
- 若某段說明是執行該 step 不可缺少的控制資訊，可以保留在主 SOP；否則應留在對應資源檔。
- 當發現此類 duplication 時，應優先重寫 step 或刪除冗餘說明，而不是同時維護兩份規格。

## Good Example

- 這個例子是好的，因為主 SOP 只交代讀取時機、產出位置與交付動作，把細部規格留在 rule 與 template。

````md
## Phase 2 -- 提交類別圖提案

1. WRITE 若需要產出類別圖提案，先讀取 `rules/類別圖提案需呈現最小必要設計資訊.md`、`templates/class-diagram-proposal.md` 與 `templates/class-diagram-proposal.example.md`，再依其要求產出提案並寫入 `specs/plan/<NN>-<本次計劃名稱>/class-diagram-proposal.md`。
````

## Bad Example

- 這個例子是壞的，因為 step 已經委派 rule 與 template，卻又把那些資源內的規格整段抄回主 SOP。

````md
## Phase 2 -- 提交類別圖提案

1. WRITE 先讀取 `rules/類別圖提案需呈現最小必要設計資訊.md`、`templates/class-diagram-proposal.md` 與 `templates/class-diagram-proposal.example.md`，再產出提案；內容需包含 Mermaid `classDiagram`、固定變更圖例、`<<new>>` / `<<modified>>` stereotype、圖外刪除項與類別職責補充說明。
````
