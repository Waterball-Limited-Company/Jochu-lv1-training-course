# Rule 1 - 編排前必須先選定改動層級

- Level: `MUST`
- 進入編排提案前，必須先判斷本次改動應屬 `local patch`、`step rewrite`、`skill rewrite` 或 `chain rewrite`。
- 改動層級的判斷應建立在 Root Cause Gate 收斂出的根因上，而不是建立在「哪種改法比較省事」。
- 不可在根因仍不明確時，就先把所有修改壓成局部修補。

## Good Example

- 這個例子是好的，因為它先根據根因決定改動層級，再進行後續編排。

````md
2. THINK 先根據 Root Cause Gate 的收斂結果，判斷本次改動應屬 local patch、step rewrite、skill rewrite 或 chain rewrite，再逐 step 決定具體的保留、重寫、展開或刪除動作。
````

## Bad Example

- 這個例子是壞的，因為它預設所有情況都只能做局部修補。

````md
2. THINK 不管 root cause 是什麼，先用 local patch 修修看。
````

# Rule 2 - 根因落在控制平面或 skill chain 拓樸時，應優先重寫或刪除，而不是疊加修補

- Level: `SHOULD`
- 若根因在 SOP phase 順序、gate 設計、step 邊界，或多個 skill 的 handoff 拓樸，應優先考慮 `step rewrite`、`skill rewrite` 或 `chain rewrite`，必要時刪除舊 phase、舊 step 或舊模組。
- 不應因為既有檔案已經存在，就預設它們都必須被保留。
- 真正需要保留的，是使用者意圖與正確協作邏輯，而不是既有檔案清單。

## Good Example

- 這個例子是好的，因為它先承認控制平面錯了，再決定重寫與刪除。

````md
Root Cause: optimize lane 缺少 Root Cause Gate。

決策：
- Change Scope: `skill rewrite`
- 刪除舊版直接從問題跳到編排提案的 phase
- 以新的雙路徑 SOP 取代原流程
````

## Bad Example

- 這個例子是壞的，因為它明知控制平面錯了，還是只敢在外層疊一條補丁式提醒。

````md
Root Cause: optimize lane 缺少 Root Cause Gate。

決策：
- 不改 phase，先在原本 step 後面補一句「必要時先分析」。
````

# Rule 3 - 被新設計淘汰的 phase、step、rule、template、script 必須清理

- Level: `MUST`
- 當新的控制平面或模組設計已淘汰舊 phase、舊 step、舊 `rules/`、舊 `templates/` 或舊 `scripts/`，必須同步刪除或清理，避免留下孤兒檔案與誤導性資源。
- 不可讓 `SKILL.md` 已經不再引用的模組繼續躺在目錄裡，讓未來維護者誤以為它們仍是有效規格。
- 若某個檔案只是為了兼容舊設計而存在，應先確認新設計是否仍需要它；若不需要，就把它列入刪除候選。

## Good Example

- 這個例子是好的，因為它把重建控制平面與清理舊資源視為同一件工程工作。

````md
4. WRITE 依確認版 plan 實際更新目標 skill 或 skill chain 的 `SKILL.md` 與相關模組，必要時刪除已被新設計淘汰的 phase、step、rule、template、script 與孤兒檔案，避免只做疊加修補。
````

## Bad Example

- 這個例子是壞的，因為它只新增新檔，卻把舊設計留下來混淆後續維護。

````md
4. WRITE 新增新的 Root Cause template，但保留已不再被引用的舊版提案檔，之後再說。
````

# Rule 4 - 既有 artifact 仍符合新邊界時，可以沿用，不必為重寫而重寫

- Level: `MAY`
- 若某個既有 `rule`、`template` 或 `script` 在新的控制平面下，責任仍單一、掛接時機仍正確、內容也不會誤導，就可以保留沿用。
- 刪除與重寫的目的，是清除錯誤拓樸與無效模組，而不是追求表面上的「全都重來」。
- 只有在既有 artifact 已與新設計的責任邊界衝突、或繼續保留會讓人誤判流程時，才應優先重寫或刪除。

## Good Example

- 這個例子是好的，因為它在敢刪之外，也保留了對仍然正確資源的節制。

````md
既有 `scripts/analyze_skill_references.py` 仍符合單一自動化目的，且新 SOP 仍會在 Phase 1 與 Phase 5 使用它，因此保留沿用。
````

## Bad Example

- 這個例子是壞的，因為它把「敢改」誤解成「一律重寫所有東西」。

````md
不管既有 artifact 是否仍然正確，全部刪掉重寫一次，這樣最乾淨。
````

# Rule 5 - 主 SOP 若重述已委派資源內容，應判為控制平面 duplication

- Level: `MUST`
- 若 duplication 發生在主 SOP step 重述已委派 `rule`、`template`、`script` 或其他 skill 的內容，應先判定它是控制平面問題，而不是單純 wording 潤飾。
- 若 duplication 侷限於少數 step，且 phase 順序與 gate 仍正確，優先判為 `step rewrite`；若 duplication 橫跨多個 phase 或 lane，優先判為 `skill rewrite`。
- 不可只在原 step 外層補一句提醒、追加平行說明，或再新增一份平行規則，就把重複內容留在原地。
- 若重複內容已無獨立價值，應直接刪除；若真正單一溯源放錯地方，應先搬移到對應資源，再重寫主 SOP。

## Good Example

- 這個例子是好的，因為它把 duplication 判成控制平面問題，並依影響範圍收斂成 `step rewrite`。

````md
觀察：
- `Phase 1 / Step 3` 已要求讀取 `rules/收斂類別與關係時的判準.md`
- step 內又重述沿用 / 新增 / 修改 / 刪除與關係判準

決策：
- Change Scope: `step rewrite`
- 刪除 step 內重述內容
- 讓判準只留在 `rules/收斂類別與關係時的判準.md`
````

## Bad Example

- 這個例子是壞的，因為它看見 duplication，卻只在原 step 外層再補一句提醒，沒有真正清理重複規格。

````md
觀察：
- step 已要求讀取對應 rule
- step 內仍有一整段重述規則內容

決策：
- 不改 step，只在句尾補上「請依 rule 執行」
````
