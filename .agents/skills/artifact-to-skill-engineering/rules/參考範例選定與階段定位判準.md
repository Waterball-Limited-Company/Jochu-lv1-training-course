# Rule 1 - 每次工程化只保留一份主要 example 作為單一主要參照

- Level: `MUST`
- 每次執行都必須收斂出一份唯一的主要 example，作為後續客製化調整、template 抽取與 target skill 反推的共同基礎。
- 若同時存在多份候選 artifact，應先判斷哪一份最貼近本次要工程化的最終樣貌；未收斂前，不可平行混用多份 example 當來源。
- 除非使用者明確要求合併兩份 example，否則預設只能有一份主要參照物。

## Good Example

- 這個例子是好的，因為它先收斂出唯一主要 example，再以它帶動後續所有階段。

````md
觀察到以下候選檔案：
- `templates/spec.example.md`
- `specs/001-online-pvp-1a2b/spec.md`

決策：
- 主要 example: `templates/spec.example.md`
- `specs/001-online-pvp-1a2b/spec.md` 只作為補充驗證，不作為平行來源
````

## Bad Example

- 這個例子是壞的，因為它同時混用多份 example，後續無法維持單一溯源。

````md
主要參考：
- `templates/spec.example.md`
- `specs/001-online-pvp-1a2b/spec.md`
- 另一份網路上找到的範例

之後綜合感覺一起改。
````

# Rule 2 - 目前階段必須以實際產物狀態判定，而不是以使用者想像的流程名稱判定

- Level: `MUST`
- 應先檢查目前實際存在什麼產物，再判定目前位於哪個階段，例如：尚未選定 example、example 未定稿、template 未抽出、target skill 未工程化、或已進入最終驗證。
- 階段判斷應停在「最早尚未完成的 gate」，而不是直接跳到使用者口頭提到的下一步。
- 若某個上游產物仍不穩定，必須回到該階段處理，不可假裝已經完成。

## Good Example

- 這個例子是好的，因為它依據實際檔案狀態判定目前還停在 example 定稿階段。

````md
觀察：
- 已有 `spec.example.md`
- 尚未明確取得「使用者確認定稿」
- template 尚未抽出

判定：
- 目前階段 = `example-finalization`
- 下一步不是做 skill engineering，而是先完成 example 定稿
````

## Bad Example

- 這個例子是壞的，因為它忽略上游產物仍未穩定，直接跳去後面階段。

````md
觀察：
- example 還在修改中

判定：
- 目前階段 = `skill-engineering`
- 先開始寫 target skill，再回頭補 example
````

# Rule 3 - 目標 skill 名稱與目錄必須在進入 template 抽取前收斂

- Level: `SHOULD`
- 在開始抽取 template 前，應已收斂 target skill 名稱與目錄，避免 template 生成後還要搬移或重命名整條鏈。
- 若使用者尚未明確命名，可先提出一個最貼近 artifact 主題的建議名稱，待使用者確認。
- 只有在使用者明確要求先不命名、且不影響後續產物放置時，才可暫時延後。

## Good Example

- 這個例子是好的，因為它先對齊目標 skill 名稱，後續 template 與 skill 工程化都對著同一個目錄進行。

````md
建議目標 skill：`artifact-to-skill-engineering`

確認後：
- target skill dir = `.agents/skills/artifact-to-skill-engineering`
- 後續 template 與 rules 都放在同一路徑下
````

## Bad Example

- 這個例子是壞的，因為它先抽模板，之後才臨時決定 skill 名稱，導致產物路徑不穩定。

````md
先把 template 抽出來放哪裡都可以，
之後再看看 skill 叫什麼名字。
````
