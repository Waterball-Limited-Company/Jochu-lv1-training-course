# Rule 1 - example 未定稿前不得進入 template 抽取

- Level: `MUST`
- 只有當 example 的結構、欄位語意、內容粒度與客製化方向都已對齊使用者預期時，才能視為通過定稿閘門。
- 若 example 仍在調整章節結構、欄位邊界、範例深度或內容取捨，表示它仍未定稿，不可提前進入 template 抽取。
- 「看起來差不多」不足以構成定稿；應以使用者明確確認或已無阻塞性差異為準。

## Good Example

- 這個例子是好的，因為先完成 example 的使用者確認，之後才開始抽 template。

````md
本輪已完成：
- 把 FR / NFR 調整為掛在 User Story 底下
- 把剩餘跨故事需求移到全域需求

使用者確認：
- 「以上就是我想要做出的改變。」

決策：
- example 已定稿
- 可以進入 template 抽取
````

## Bad Example

- 這個例子是壞的，因為 example 還在改，卻直接開始抽 template。

````md
目前還不確定全域需求要不要保留 NFR，
但先把 template 抽出來，之後再一起改。
````

# Rule 2 - 高影響歧義才應升級 /clarify，低影響調整應直接在 example 內迭代

- Level: `MUST`
- 若歧義會改變 artifact 結構、欄位定義、需求邊界、核心流程、模板槽位或後續 skill 的推理邏輯，應升級 `/clarify`。
- 若差異只影響 wording、例句細節、敘事語氣、示範內容輕微增減或其他可安全直接修改的低影響問題，應直接在 example 內迭代，不必為問而問。
- 升級 `/clarify` 的目的是清除真正會影響後續抽象化與工程化的缺口，不是把所有微調都改寫成問答。

## Good Example

- 這個例子是好的，因為它只在結構邊界真的不清楚時才升級 clarify。

````md
待確認問題：
- FR / NFR 是否應直接掛在 User Story 底下
- 全域需求是否只保留跨故事需求

影響：
- 會直接改變 artifact 結構與後續 template 抽取方式

決策：
- 升級 `/clarify`
````

## Bad Example

- 這個例子是壞的，因為它把低影響 wording 微調也升級成 clarify。

````md
待確認問題：
- 標題要寫「功能規格」還是「產品規格」

影響：
- 只影響 wording

決策：
- 先跑 `/clarify`
````

# Rule 3 - 本輪 example 調整應以最小必要改動為主

- Level: `SHOULD`
- 每一輪都應優先提出最小必要修改，使 example 逐步逼近目標樣貌，而不是一次重寫所有內容。
- 若已能從少量結構性調整驗證使用者意圖，應先完成該輪再看是否需要下一輪，而不是預先展開過多優化建議。
- 這樣能降低誤改風險，也讓後續 template 抽取更容易對齊最終版本。

## Good Example

- 這個例子是好的，因為它先只改最關鍵的階層結構，等使用者確認後再往下走。

````md
本輪調整：
- 把 FR / NFR 搬到各 User Story 底下
- 把故事外需求改名為全域需求

暫不處理：
- 其他 wording 微調
- 額外新增欄位
````

## Bad Example

- 這個例子是壞的，因為它在尚未確認核心結構前，就一次加入大量額外改動。

````md
本輪調整：
- 改 artifact 結構
- 改命名風格
- 改編號策略
- 新增三個新欄位
- 重寫所有說明文字
````
