# Rule 1 - 最終 technical-research 必須寫到固定路徑

- Level: `MUST`
- `technical-research` 的最終產物路徑固定為 `specs/<NNN-plan-package>/system-analyze/technical-research.md`。
- 不可把完成稿只留在聊天訊息、暫存筆記或其他任意檔名中，否則後續系統分析與一致性檢查難以穩定接續。
- 若 `system-analyze/` 目錄尚不存在，應先建立該目錄再寫入 `technical-research.md`。

## Good Example

- 這個例子是好的，因為輸出路徑與 skill 契約完全對齊。

```text
specs/001-photo-albums/system-analyze/technical-research.md
```

## Bad Example

- 這個例子是壞的，因為它把技術決策放到臨時位置，其他流程無法穩定接手。

```text
tmp/tech-choices.md
```

# Rule 2 - 最終 plan 必須寫到 package 根目錄固定路徑

- Level: `MUST`
- `plan` 的最終產物路徑固定為 `specs/<NNN-plan-package>/plan.md`（與 `spec.md` 同層），**不可**寫在 `system-analyze/` 底下。
- 必須在同 package 的 `technical-research.md` 已寫入後再產出 `plan.md`。
- 若 package 目錄尚不存在，應先確認上游 `spec.md` 的 package，再寫入 `plan.md`。

## Good Example

- 這個例子是好的，因為 plan 在 package 根目錄，與 research 分層清楚。

```text
specs/001-photo-albums/plan.md
specs/001-photo-albums/system-analyze/technical-research.md
```

## Bad Example

- 這個例子是壞的，因為把 plan 誤放進 system-analyze。

```text
specs/001-photo-albums/system-analyze/plan.md
```

# Rule 3 - `plan-package` 必須與上游 spec 目錄對齊

- Level: `MUST`
- `plan-package` 必須沿用同功能的 `specs/<NNN-plan-package>/` 目錄名，採 `NNN-` 三位數前綴加英文小寫 kebab-case slug。
- research 與 plan 必須寫入同一 `plan-package`，不可另開平行目錄。

## Good Example

- 這個例子是好的，因為 spec、research、plan 落在同一 package。

```text
specs/001-photo-albums/spec.md
specs/001-photo-albums/plan.md
specs/001-photo-albums/system-analyze/technical-research.md
```

## Bad Example

- 這個例子是壞的，因為產物分散在不同 package。

```text
specs/001-photo-albums/spec.md
specs/002-plan-only/plan.md
```

# Rule 4 - research 與 plan 的標題 metadata 必須對齊契約

- Level: `MUST`
- research 的 H1 固定為 `# 技術可行性研究：{{FEATURE_NAME}}`；plan 的 H1 固定為 `# 系統分析：{{FEATURE_NAME}}`。功能中文名與同 package 的 `spec.md` 對齊。
- 兩份產物的小標皆固定三行、分行書寫：`**功能分支**:`、`**建立日期**:`、`**狀態**:`；功能分支值必須以 inline code 包住，且與所在 `plan-package` 相同。
- `**建立日期**:` 使用本次產出日期（`YYYY-MM-DD`）；`**狀態**:` 第一版預設 `草稿`。
- 不可再使用已淘汰的舊標題（例如把 plan 寫成「實作計畫」，或把「技術可行性研究」寫成其他別名），也不可使用單行 pipe metadata。

## Good Example

- 這個例子是好的，因為兩份產物標題契約清楚且可對回目錄。

```md
# 技術可行性研究：照片日期相簿整理

**功能分支**: `001-photo-albums`
**建立日期**: 2026-07-21
**狀態**: 草稿
```

```md
# 系統分析：照片相簿整理

**功能分支**: `001-photo-albums`
**建立日期**: 2026-07-21
**狀態**: 草稿
```

## Bad Example

- 這個例子是壞的，因為沿用舊 pipe metadata，且功能欄與目錄名脫節。

```md
# Tech Research

**功能**: photo-db-final-v2 ｜ **日期**: 2026-07-21 ｜ 來源：聊天紀錄
```
