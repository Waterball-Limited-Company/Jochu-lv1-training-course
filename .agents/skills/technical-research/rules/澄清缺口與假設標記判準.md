# Rule 1 - 高影響缺口必須先澄清，或在正文標 `[NEEDS CLARIFICATION]`

- Level: `MUST`
- 若缺口會改變整體架構邊界（如前後端分離 vs 單體）、主技術棧（執行環境／主要框架）、或讓多數決策同時改寫，優先呼叫 `/clarify`；未拍板前不可寫成看似已確認的 Decision。
- 若為完成可 Review 產物而必須先寫暫定內容（例如供父 skill 硬停審閱），必須在受影響的 Decision／條列正文內嵌 `[NEEDS CLARIFICATION: …]`，並可在檔末 `## 假設` 註明第一版暫定取捨；不可只靠假設、正文卻寫死。
- 標記密度對齊 specify：只標高影響缺口；低風險預設不標標記。

## Good Example

- 這個例子是好的，因為高影響未決在正文可見，且未假裝已定案。

```md
- **Decision**: 自動成冊粒度暫定「日」[NEEDS CLARIFICATION: 需確認自動成冊以日、月或年為單位]
```

## Bad Example

- 這個例子是壞的，因為主結構未定卻寫成已確認決策且無標記。

```md
先假設一定是 React Native，把五則決策都寫完，之後有空再問。
```

# Rule 2 - `[NEEDS CLARIFICATION]` 格式必須與 specify 一致

- Level: `MUST`
- 待澄清標記固定為 `[NEEDS CLARIFICATION: …]`（`NEEDS` 複數），必須明說缺的決策軸，不可寫成空泛提醒。
- 不可使用 `[Need Clarification]`、`[NEED CLARIFICATION]` 等變體作為契約標記。

## Good Example

- 這個例子是好的，因為決策軸清楚且字串與 specify 一致。

```md
[NEEDS CLARIFICATION: 需確認第一版要桌面原生 App 還是本機 Web]
```

## Bad Example

- 這個例子是壞的，因為字串不一致且語意含糊。

```md
[Need Clarification: 之後再想]
```

# Rule 3 - 低風險不確定性只寫檔末「假設」，不打待澄清標記

- Level: `MUST`
- 若缺口不致改變 technical-research 主結構，可用第一版假設繼續，寫入檔末 `## 假設`（`- ` 條列），格式與 `spec.md` 相同。
- 此類低風險項目**不要**使用 `[NEEDS CLARIFICATION]`，以免稀釋高影響標記。
- 澄清後必須回寫 `## 假設` 與相關決策；不要另開缺口追蹤表格章節。

## Good Example

- 這個例子是好的，因為低風險只進假設。

```md
## 假設

- 開發期前後端分 port 時可選用 cors；非強制選型
```

## Bad Example

- 這個例子是壞的，因為把低風險偏好也打成待澄清標記。

```md
[NEEDS CLARIFICATION: cors 套件要不要加]
```

# Rule 4 - 檔案最下方必須有與 spec.md 同格式的「假設」區塊

- Level: `MUST`
- `technical-research.md` 必須在檔案最下方（所有決策之後）提供 `## 假設`，以 `- ` 條列全域前提與本檔第一版設計假設。
- 對已標 `[NEEDS CLARIFICATION]` 且採暫定往前的項目，應在假設中交代暫定取捨；不可省略本章節。

## Good Example

- 這個例子是好的，因為假設在檔末且可對應標記。

```md
## 假設

- 自動成冊粒度正文已標 NEEDS CLARIFICATION；第一版暫定以「日」實作
```

## Bad Example

- 這個例子是壞的，因為缺少檔末假設區塊。

```md
## 決策 5: ...

（檔案結束）
```

# Rule 5 - 上游 `[NEEDS CLARIFICATION]` 必須被消化，不可默默消失

- Level: `SHOULD`
- 對 `spec.md` 中與技術選型相關的 `[NEEDS CLARIFICATION]`，應在本輪澄清、在本檔正文保留／改寫對應標記、寫入 `## 假設`，或在假設聲明「本期不影響技術決策主結構／延後」。
- 不可略過上游缺口，讓讀者以為 technical-research 已完全定案。

## Good Example

- 這個例子是好的，因為上游缺口都有去向。

```md
spec 缺口 A → 已 clarify
spec 缺口 B → 本檔正文保留 [NEEDS CLARIFICATION: …] 並於 ## 假設註暫定
spec 缺口 C → ## 假設 註明不影響本期技術主結構
```

## Bad Example

- 這個例子是壞的，因為上游缺口被默默丟掉。

```md
只寫有把握的套件，spec 裡的 NEEDS CLARIFICATION 一律不提。
```
