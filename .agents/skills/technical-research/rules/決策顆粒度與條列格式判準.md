# Rule 1 - 以功能／技術決策為主要顆粒度

- Level: `MUST`
- 產物正文以 `## 決策 N: <簡短標題>` 為一級單位；`N` 從 1 開始連續編號。
- 每一則決策應對準一個可 Review 的技術選擇（架構、某技術面工具鏈、某關鍵能力實作手段、套件組合或存放策略），不可把整份研究壓成單一巨無霸決策。
- 也不可切得過碎（例如每個套件各開一則、卻沒有共同決策軸）；同一決策軸的套件組合可收在同一則。

## Good Example

- 這個例子是好的，因為決策軸清楚，且粒度可 Review。

```md
## 決策 1: 採用極簡 Web 前後端分離架構
## 決策 3: 主頁相簿重排使用原生 Drag and Drop
## 決策 4: 後端使用最小必要套件處理上傳、EXIF 與縮圖
```

## Bad Example

- 這個例子是壞的，因為要嘛過粗、要嘛過碎。

```md
## 決策 1: 全部技術一次講完

或

## 決策 1: 選 express
## 決策 2: 選 multer
## 決策 3: 選 sharp
（沒有共同決策軸）
```

# Rule 2 - 每則決策必須含 Decision／Rationale／Alternatives considered

- Level: `MUST`
- 每一則 `## 決策 N` 底下必須以條列提供且僅能用這三個固定標籤：
  - `- **Decision**: ...`
  - `- **Rationale**:`（其下可用子條列說明）
  - `- **Alternatives considered**:`（其下必須有至少一個被排除方案；每條寫成 `方案：不選原因`，冒號兩側皆有非空白文字）
- 不可省略任一標籤，也不可改成其他同義標題（例如「選擇」「理由」「其他方案」）而失去固定契約。
- Decision 應寫出所選技術／做法；Rationale 說明為何；Alternatives 對照未選項。

## Good Example

- 這個例子是好的，因為三欄齊全且以條列呈現。

```md
## 決策 2: 前端不引入 runtime framework，只以 Vite 作為開發／建置工具

- **Decision**: 使用 `Vite` 做開發伺服與建置；執行期不引入 React／Vue
- **Rationale**:
  - 核心互動可用原生 ES modules 完成
  - 減少學習曲線
- **Alternatives considered**:
  - 狀態管理函式庫：單頁小應用價值低
  - UI component library：削弱原生基礎教學聚焦
```

## Bad Example

- 這個例子是壞的，因為缺欄或標籤不一致。

```md
## 決策 2: 前端工具

選 Vite。理由很多。其他的我就不寫了。
```

# Rule 3 - 以條列為主，避免長篇敘事取代決策結構

- Level: `SHOULD`
- 決策正文應以條列表達；需要補充時用短句，不要改寫成無結構長文。
- 技術關鍵字可用 inline code 標示（如 `Express`、`better-sqlite3`），但不要把整段 Decision 變成散文。
- 檔末 `## 假設` 同樣使用 `- ` 條列，不另開散文段落取代。

## Good Example

- 這個例子是好的，因為資訊密度高且可掃讀。

```md
- **Decision**: 核心套件採用 `express`、`multer`、`better-sqlite3`、`exifr`、`sharp`
- **Rationale**:
  - `multer`：簡化 multipart 上傳
  - `exifr`：解析可用日期
```

## Bad Example

- 這個例子是壞的，因為變成難以 Review 的長文。

```md
關於後端我們經過深思熟慮之後覺得也許可以考慮很多套件然後...
```
