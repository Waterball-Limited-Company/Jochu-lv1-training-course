# Rule 1 - 僅可獨立到達的全頁算「頁面」

- Level: `MUST`
- 「頁面」只包含使用者可獨立到達的全頁／主畫面；Modal、Drawer、確認框、本機檔案選取器、頁內選目標清單等不算獨立頁。
- 附屬 UI 應寫進所屬頁的「操作 Flow」與「呈現內容」，不可升級成 `## 頁面：`。
- 導覽表只描述全頁之間的轉移。

## Good Example

- 這個例子是好的，因為建立表單留在主頁 Flow，沒有另開頁。

```md
## 頁面：主頁（Home）
### 操作 Flow
使用者輸入名稱並確認建立（Modal／表單附屬本頁）
```

## Bad Example

- 這個例子是壞的，因為把 Modal 升格成頁面。

```md
## 頁面：建立相簿 Dialog（CreateAlbumModal）
```

# Rule 2 - 每個頁面必須展開五塊合約內容

- Level: `MUST`
- 每個 `## 頁面：名稱（代號）` 必須依序包含：`### 職責`、`### 呈現內容`、`### 操作 Flow`、`### 導覽`、`### API 對應`。
- **職責**：條列此頁處理的 US／FR（可標入口／驗證端）。
- **呈現內容**：此頁顯示哪些資訊／區塊／空狀態等（人機「看見什麼」）。
- **操作 Flow**：頁內使用者↔系統／API 的 Mermaid Sequence；空／錯／載入等狀態寫進呈現或 Flow，不另開「狀態」專章。
- **導覽**：表格列出操作 → 前往頁面。
- **API 對應**：表格列出使用者操作 → API → 說明；頁內需要的列表資料來源不可省略。

## Good Example

- 這個例子是好的，因為五塊齊全，且分配目標有 API 對應。

```md
## 頁面：照片庫頁（Photo Library）
### 職責
- **US-4**：…
### 呈現內容
- 分配目標相簿清單，來自 `GET /albums`
### 操作 Flow
（含 GET /albums 再 assign）
### 導覽
| 操作 | 前往頁面 |
### API 對應
| 載入分配目標相簿清單 | `GET /albums` | … |
```

## Bad Example

- 這個例子是壞的，因為缺呈現內容，且操作沒有 API 對應。

```md
## 頁面：照片庫頁（Photo Library）
### 職責
- US-4
### 導覽
| 返回 | 主頁 |
```

# Rule 3 - UI-Plan 是人機介面合約，不是視覺 Design

- Level: `MUST`
- 不寫色票、字級、間距、動畫、元件庫實作細節或像素級 mockup。
- 可寫資訊架構、操作意圖、導覽、狀態意圖，以及與 API／資料欄位的對應關係。
- 若需要示意版面，最多用文字描述區塊；不把 Figma／設計稿當成本 skill 產物。

## Good Example

- 這個例子是好的，因為描述的是合約資訊而非視覺。

```md
### 呈現內容
- 依 group_date 分組的相簿列表；項目含 name、source、photo_count
```

## Bad Example

- 這個例子是壞的，因為滑進視覺 Design。

```md
### 呈現內容
- 主色 #6C5CE7，卡片圓角 16px，標題用 Inter 24/Bold
```

# Rule 4 - 檔案必須提供頁面總覽以串起導覽關係

- Level: `MUST`
- 在各頁展開之後，必須有 `## 頁面總覽（導覽關係）`，含導覽關係圖（Mermaid flowchart）與「頁面｜主要 US」表。
- 總覽應與各頁導覽表一致，不可出現總覽有、內文無的頁面。

## Good Example

- 這個例子是好的，因為總覽與內文頁面一致。

```md
## 頁面總覽（導覽關係）
（flowchart：Home ↔ Album、Home ↔ Library）
| 頁面 | 主要 US |
| 主頁 | US-3；入口承載 US-1、US-5 |
```

## Bad Example

- 這個例子是壞的，因為總覽多出未展開的頁面。

```md
| 設定頁 | US-99 |
（全文沒有 ## 頁面：設定頁）
```
