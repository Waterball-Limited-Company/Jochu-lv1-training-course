# Rule 1 - 只有穩定且可重複使用的輸出格式才應抽成樣板

- Level: `MUST`
- 應抽成樣板的內容，必須是用來指導 AI 生成檔案內容格式的固定結構、欄位順序、語法骨架或可重複替換的填位符號。
- 若某段內容描述的是品質判準、限制、禁止事項或常見失誤，應抽成 RuleFile，而不是樣板。
- 若某段內容描述的是 phase 順序、必要動作、交付物產出或流程前後關係，應留在原 step 中，不應抽離成樣板。

## Good Example

- 這個例子是好的，因為它把可重複使用的 Mermaid 輸出骨架抽成樣板，而把流程動作留在主 SOP。

```md
適合抽成樣板：
- `templates/class-diagram.mmd`
- `templates/class-diagram.example.mmd`
- 用來固定 `classDiagram` 的語法骨架、類別區塊與填位符號。
```

## Bad Example

- 這個例子是壞的，因為它把規則與流程一起塞進樣板，樣板不再只是格式資源。

```md
不適合抽成樣板：
- 「優先保留最小必要類別」
- 「避免過度設計」
- 「先盤點類別，再整理關係，再輸出圖」
```

# Rule 2 - 原 step 必須保留最小可執行動作

- Level: `MUST`
- 展開樣板後，原 step 仍必須讓執行者知道當下要生成什麼產物、要對哪份輸入進行改寫，以及這一步的直接目的。
- 樣板只補充輸出格式，不取代 step 本身的動詞、處理對象與產物目的。
- 若移除樣板路徑後，step 會變成只剩「去看某個 template」而失去行動指向，表示抽離邊界過度。

## Good Example

- 這個例子是好的，因為 step 仍然清楚要求產出類別圖，只是補上需要讀取的樣板資源。

```md
2. WRITE 若需要產出類別圖，先讀取 `templates/class-diagram.mmd` 與 `templates/class-diagram.example.mmd`，再依目前分析結果填入類別、欄位與關係。
```

## Bad Example

- 這個例子是壞的，因為 step 只剩讀檔提醒，沒有交代當下要完成的產物與動作。

```md
2. WRITE 若有需要，讀取相關樣板。
```

# Rule 3 - 優先以單一產物格式收斂一組樣板雙檔

- Level: `SHOULD`
- 同一組骨架檔與範例檔應盡量圍繞單一產物格式或單一輸出主題，避免把多種無關格式混在一起。
- 若同一個 step 需要多種不同格式的輸出，應優先考慮拆成多組樣板雙檔，而不是把所有內容塞進同一份骨架。
- 樣板主題越單一，執行者越容易直接複製、替換與比對範例。

## Good Example

- 這個例子是好的，因為一組樣板只處理 Mermaid 類別圖，不混入其他格式。

```text
templates/class-diagram.mmd
templates/class-diagram.example.mmd
```

## Bad Example

- 這個例子是壞的，因為同一組樣板同時想承載圖表與說明文件，主題過散。

```text
templates/architecture-output.md
templates/architecture-output.example.md
# 內容同時混入 Mermaid 類別圖、API 規格與會議摘要格式
```
