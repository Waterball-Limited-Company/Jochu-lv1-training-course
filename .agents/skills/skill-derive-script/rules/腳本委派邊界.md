# Rule 1 - 只有可重複且輸入輸出明確的職責才應委派給 Python script

- Level: `MUST`
- 適合抽成 Python script 的職責，必須是可重複執行、輸入輸出明確，且主要價值在自動化處理而非開放式判斷。
- 若某個 step 的核心價值仍是需求理解、設計取捨、語意判讀或與使用者協商，該核心責任應留在原 step，不應整段外包給 script。
- 不可把「這一步很麻煩」當成唯一理由就展開成 script，必須能說清楚 script 接收什麼、產出什麼、替原 step 節省哪段重複操作。

## Good Example

- 這個例子是好的，因為委派出去的是結構化轉換工作，輸入與產出都能穩定定義。

```md
2. THINK 若需要把多份訪談逐字稿整理成固定 JSON 結構，判斷是否委派給 Python script 處理欄位抽取與格式正規化，而把最終欄位判讀標準留在原 step。
```

## Bad Example

- 這個例子是壞的，因為它把需要高層判斷的設計決策整段外包給 script，主流程失去核心思考責任。

```md
2. THINK 若要決定整個系統該採用哪種架構，直接委派給 Python script 幫我做設計判斷。
```

# Rule 2 - 展開 script 後，原 step 仍必須保留最小可執行動作與目的

- Level: `MUST`
- 展開 script 後，原 step 仍必須讓執行者知道當下為何要執行這支 script，以及執行完要把結果接回哪個流程。
- script 只承接可自動化的局部職責，不取代原 step 的動詞、處理對象與階段目的。
- 若移除 script 路徑後，原 step 只剩下「去跑某支 script」而沒有說明當下要完成什麼，表示委派邊界過度。

## Good Example

- 這個例子是好的，因為即使不看 script 細節，step 仍清楚交代執行腳本的目的與後續用途。

```md
1. DELEGATE 若需要先把原始事件紀錄整理成後續分析可用的 JSON，執行對應 Python script 產出標準化結果，再用該結果繼續後續判讀。
```

## Bad Example

- 這個例子是壞的，因為 step 只剩下執行指令，沒有交代為何執行與如何接回流程。

```md
1. DELEGATE 執行 `scripts/process.py`。
```

# Rule 3 - 優先從重複操作、格式轉換與機械檢查收斂 script 主題

- Level: `SHOULD`
- 在判斷是否值得抽成 script 時，應優先檢查該 step 是否包含重複操作、結構化格式轉換、批次處理或機械式檢查。
- 若某段工作主要困難在於一次性的上下文理解，而不是可重播的流程，自動化價值通常較低。
- 同一支 script 應盡量圍繞單一自動化主題，避免把資料整理、外部同步、通知發送與決策判斷混寫在一起。

## Good Example

- 這個例子是好的，因為它把「批次重命名與 frontmatter 修正」收斂成單一可重播主題。

```md
適合展開成 script：批次掃描文章檔案、修正 frontmatter 欄位名稱、輸出修正摘要。
```

## Bad Example

- 這個例子是壞的，因為它把多種彼此無關的責任綁在同一個委派判斷裡，主題過散。

```md
適合展開成 script：順便決定命名策略、改寫文章內容、上傳雲端、寄信通知、整理會議結論。
```
