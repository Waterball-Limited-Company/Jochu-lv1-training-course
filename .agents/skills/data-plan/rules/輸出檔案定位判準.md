# Rule 1 - 最終 data-plan 必須寫到固定路徑

- Level: `MUST`
- `data-plan` 的最終產物路徑固定為 `specs/<NNN-plan-package>/system-analyze/data-plan.md`。
- 不可把完成稿只留在聊天訊息、暫存筆記或其他任意檔名中，否則後續系統分析與一致性檢查難以穩定接續。
- 若 `system-analyze/` 目錄尚不存在，應先建立該目錄再寫入 `data-plan.md`。

## Good Example

- 這個例子是好的，因為輸出路徑與 skill 契約完全對齊。

```text
specs/001-photo-albums/system-analyze/data-plan.md
```

## Bad Example

- 這個例子是壞的，因為它把資料合約放到臨時位置，其他流程無法穩定接手。

```text
tmp/schema.md
```

# Rule 2 - `plan-package` 必須與上游 spec 目錄對齊

- Level: `MUST`
- `plan-package` 必須沿用同功能的 `specs/<NNN-plan-package>/` 目錄名，採 `NNN-` 三位數前綴加英文小寫 kebab-case slug。
- 若使用者已指定既有 package，應寫入該 package 的 `system-analyze/`，不可另開平行目錄。
- 若尚無 package，應先確認上游 `spec.md` 的 package，再建立對應的 `system-analyze/`。

## Good Example

- 這個例子是好的，因為 data-plan 與 spec 落在同一 package。

```text
specs/001-photo-albums/spec.md
specs/001-photo-albums/system-analyze/data-plan.md
```

## Bad Example

- 這個例子是壞的，因為資料合約與 spec 分散在不同 package，追溯會斷開。

```text
specs/001-photo-albums/spec.md
specs/002-data-only/system-analyze/data-plan.md
```

# Rule 3 - 標題與小標必須對齊 specify 契約

- Level: `MUST`
- H1 固定為 `# 資料計畫：{{FEATURE_NAME}}`，功能中文名與同 package 的 `spec.md` 對齊。
- 小標固定三行、分行書寫：`**功能分支**:`、`**建立日期**:`、`**狀態**:`；其中功能分支值必須以 inline code 包住，且與所在 `plan-package` 相同。
- `**建立日期**:` 使用本次產出日期（`YYYY-MM-DD`）；`**狀態**:` 第一版預設 `草稿`。
- 不可再使用單行 `**功能**｜**日期**｜來源：`，也不可在 H1 塞入實體英文名清單。

## Good Example

- 這個例子是好的，因為標題與小標與 `spec.md` 同構，且功能分支可對回目錄。

```md
# 資料計畫：照片相簿整理

**功能分支**: `001-photo-albums`
**建立日期**: 2026-07-20
**狀態**: 草稿
```

## Bad Example

- 這個例子是壞的，因為沿用舊 pipe metadata，且功能欄與目錄名脫節。

```md
# Data Plan: Album, Photo

**功能**: photo-db-final-v2 ｜ **日期**: 2026-07-20 ｜ 來源：聊天紀錄
```

# Rule 4 - 有任何 DB 持久化時必須另產 DDL.md

- Level: `MUST`
- 若 `plan.md`／`technical-research.md` 顯示會以任何資料庫（含本機嵌入式 SQLite、遠端 Postgres／MySQL 等）做持久化讀寫，必須另寫 `specs/<NNN-plan-package>/system-analyze/DDL.md`。
- 純記憶體、或純檔案／JSON／YAML 等非 DB 持久化、且明確無 DB 讀寫時，不產 `DDL.md`。
- `DDL.md` 不得併回 `data-plan.md`；`data-plan.md` 不得再內嵌 `## ERD`／`## DDL`／SQL 腳本。

## Good Example

- 這個例子是好的，因為本機 SQLite 也產獨立 DDL 檔。

```text
Storage: SQLite → 產出 data-plan.md + DDL.md
```

## Bad Example

- 這個例子是壞的，因為有 DB 卻把 ERD／SQL 塞回 data-plan 單檔。

```md
# 資料計畫：…
## ERD
## DDL
```

# Rule 5 - DDL.md 標題與對齊 metadata 固定

- Level: `MUST`
- 當產出 `DDL.md` 時，H1 固定為 `# DDL：{{FEATURE_NAME}}`，功能中文名與同 package `spec.md`／`data-plan.md` 對齊。
- 小標至少含：`**功能分支**:`、`**建立日期**:`、`**狀態**:`、`**對齊**: \`system-analyze/data-plan.md\``。
- 功能分支值以 inline code 包住，且與 `plan-package` 相同。

## Good Example

- 這個例子是好的，因為 DDL 檔可追溯回 data-plan。

```md
# DDL：照片相簿整理應用程式

**功能分支**: `001-photo-albums`
**建立日期**: 2026-07-22
**狀態**: 草稿
**對齊**: `system-analyze/data-plan.md`
```

## Bad Example

- 這個例子是壞的，因為缺少對齊指向，雙檔契約斷開。

```md
# Schema
**功能**: albums
```
