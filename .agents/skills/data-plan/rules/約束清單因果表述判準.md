# Rule 1 - 約束必須用「因為…所以…」因果句

- Level: `MUST`
- 約束清單每條必須採 `因為{{原因}}，所以{{設計}}` 句型，直接說明設計因果。
- 不可使用 `C-01` 這類編號，也不可使用 `→`／`->` 箭頭串接。
- 原因側應能對回 spec 的業務／邊界需求；設計側應指向具體欄位、屬性或不落庫策略。

## Good Example

- 這個例子是好的，因為因果清楚且可落地到欄位。

```md
- 因為相簿必須可辨識（FR-001），所以 albums.name 必填且非空白。
```

## Bad Example

- 這個例子是壞的，因為用編號與箭頭，讀起來像清單而非設計理由。

```md
- C-01 albums.name 非空白 → FR-001
```

# Rule 2 - FR／US 引用緊接在「因為」子句之後

- Level: `MUST`
- 有對應 FR／US 時，編號括號必須緊接原因之後，例如 `因為相簿必須可辨識（FR-001），所以…`。
- 不可把 `(FR-001)` 放到整句句尾或只寫在「所以」之後。
- 多個編號用逗號分隔；無對應追溯時可省略括號，但仍需寫清因果。

## Good Example

- 這個例子是好的，因為追溯貼在原因上。

```md
- 因為只支援 JPG／PNG／HEIC（US1-FR3），所以 photos.mime_type 限上述三種值。
```

## Bad Example

- 這個例子是壞的，因為 FR 放錯位置。

```md
- 因為只支援三種 MIME，所以 photos.mime_type 設 CHECK (US1-FR3)。
```

# Rule 3 - 約束清單只列實體內部欄位／屬性約束

- Level: `MUST`
- 約束清單只涵蓋單一實體內部：必填／唯一／值域、衍生不落庫、單實體生命週期等。
- **不可**把實體間關聯形狀寫進約束清單（例如 1:N 對 M:N、不設中介表、無父子邊、`ON DELETE CASCADE` 理由）；那些屬 `DDL.md`「設計脈絡」。
- 可落庫的欄位約束應能在有 DB 時的 DDL `CHECK`／`NOT NULL`／唯一索引中找到對應；不落庫策略應能在衍生屬性中找到對應。
- 不要寫無法驗證的空泛口號（例如「要保持彈性」）。

## Good Example

- 這個例子是好的，因為只談欄位／屬性，並可對上驗證或衍生策略。

```md
- 因為計數不可與歸屬列不一致，所以 photo_count 不落庫，改由 photos 計數。
```

## Bad Example

- 這個例子是壞的，因為把關聯拓撲寫進約束清單。

```md
- 因為一張照片只屬一本相簿，所以不設 M:N 中介表並 ON DELETE CASCADE。
```
