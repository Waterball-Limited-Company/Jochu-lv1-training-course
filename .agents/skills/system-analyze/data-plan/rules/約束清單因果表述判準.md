# Rule 1 - 約束必須用「因為…所以…」因果句

- Level: `MUST`
- 約束清單每條必須採 `因為{{原因}}，所以{{設計}}` 句型，直接說明設計因果。
- 不可使用 `C-01` 這類編號，也不可使用 `→`／`->` 箭頭串接。
- 原因側應能對回 spec 的業務／邊界需求；設計側應指向具體表、欄位、鍵或策略。

## Good Example

- 這個例子是好的，因為因果清楚且可落地。

```md
- 因為相簿不可巢狀（FR-003），所以 albums 不提供 parent_album_id（或同等父相簿欄位）。
```

## Bad Example

- 這個例子是壞的，因為用編號與箭頭，讀起來像清單而非設計理由。

```md
- C-01 albums 無 parent_album_id → 禁止巢狀 (FR-003)
```

# Rule 2 - FR／US 引用緊接在「因為」子句之後

- Level: `MUST`
- 有對應 FR／US 時，編號括號必須緊接原因之後，例如 `因為相簿不可巢狀（FR-003），所以…`。
- 不可把 `(FR-003)` 放到整句句尾或只寫在「所以」之後。
- 多個編號用逗號分隔；無對應追溯時可省略括號，但仍需寫清因果。

## Good Example

- 這個例子是好的，因為追溯貼在原因上。

```md
- 因為同一相簿內同一照片不可重複（FR-004），所以 album_photos 以 (album_id, photo_id) 為主鍵。
```

## Bad Example

- 這個例子是壞的，因為 FR 放錯位置。

```md
- 因為同一相簿內同一照片不可重複，所以 album_photos 以 (album_id, photo_id) 為主鍵 (FR-004)。
```

# Rule 3 - 約束應覆蓋關鍵完整性，且能對上 DDL

- Level: `SHOULD`
- 至少涵蓋：禁止結構、必填／唯一、關聯基數、外鍵級聯、不落庫衍生策略、允許 null 的業務理由等會影響正確性的項目。
- 每條「所以」指出的結構，應能在 DDL 或衍生屬性策略中找到對應實作。
- 不要寫無法驗證的空泛口號（例如「要保持彈性」）。

## Good Example

- 這個例子是好的，因為約束可以直接檢查 DDL。

```md
- 因為歸屬列不可懸空，所以 album_photos.album_id / photo_id 設外鍵並 ON DELETE CASCADE。
```

## Bad Example

- 這個例子是壞的，因為約束無法對上任何結構。

```md
- 因為資料很重要，所以要好好設計。
```
