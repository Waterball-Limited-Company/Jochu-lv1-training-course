# Rule 1 - 資料實體必須從 spec 推導，決策權在 data-plan

- Level: `MUST`
- 實體、欄位、衍生屬性與持久化策略必須由 `spec.md`（及本輪 clarify 決策）推導；儲存／執行環境選型對齊 `technical-research.md`。
- 不可把其他系統分析產物（例如既有 API 合約）的資源形狀、欄位名或 JSON 範例當成資料模型真相再「對齊抄寫」。
- data-plan 是系統分析中供下游引用的持久化真相；落表粒度、中介表、欄位與不落表語意的最終決策由本 skill 萃取，不可外包給下游或照抄 spec 關鍵實體清單。

## Good Example

- 這個例子是好的，因為以 spec 為來源，由 data-plan 決定落庫欄位。

```md
FR-001 需要可辨識相簿名稱 → albums.name
FR-003 禁止巢狀 → 不提供 parent_album_id
FR-009／FR-010 拖放順序要保留 → albums.sort_order
```

## Bad Example

- 這個例子是壞的，因為拿下游合約當來源複製欄位。

```md
打開下游 API 的 Album JSON，欄位照抄成 albums 表。
```

# Rule 2 - `## 關鍵實體` 是種子與參考，不是強制一表一實體

- Level: `MUST`
- 若 `spec.md` 存在 `## 關鍵實體`，必須先盤點其中的實體名、描述與關鍵屬性，作為實體萃取的種子／參考清單。
- 種子**不強制**對應成一級資料表：data-plan 可決定落成獨立表、合併進他表、以狀態／查詢語意表達，或新增 spec 未列出的中介表（如 M:N）。
- 每個種子項目都必須有去向：採納為 `## 實體`、併入既有實體並在簡介／假設說明，或明示「不獨立落表」的理由；不可默默消失。
- 若 `## 關鍵實體` 章節不存在（功能不涉及資料或已刪除），改由 US／FR／邊界與 technical-research 推導實體，不因缺席而停工。

## Good Example

- 這個例子是好的，因為先吃種子，再由 data-plan 決定落表方式。

```md
關鍵實體「相簿」「照片」→ 一級實體 Album、Photo
關鍵實體「照片庫」→ 不另開表，以 photos.in_library 查詢語意表達
另依 M:N 需求新增 AlbumPhoto 中介實體（種子未列也可新增）
```

## Bad Example

- 這個例子是壞的，因為把關鍵實體當成強制 DDL 清單照抄。

```md
spec 寫了相簿、照片、照片庫 → 一定建三張表 libraries / albums / photos
（不問是否真需獨立落表）
```

# Rule 3 - 會落成資料表的概念都是一級「資料實體」

- Level: `MUST`
- 只要會持久化成一張表（含純關聯／中介表），就必須有對應的 `## 實體：Name（中文）` 章節。
- 不可只在 ERD／DDL 出現關聯表，卻省略實體章節。
- 實體簡介寫清職責邊界；不要另開「職責」小標。

## Good Example

- 這個例子是好的，因為 M:N 的中介表也有一級實體。

```md
## 實體：AlbumPhoto（相簿照片歸屬）

相簿與照片的多對多歸屬。...
```

## Bad Example

- 這個例子是壞的，因為關聯表只藏在 DDL。

```md
（實體只有 Album、Photo）
DDL 裡突然出現 CREATE TABLE album_photos ...
```

# Rule 4 - 欄位表必須使用固定六欄

- Level: `MUST`
- 每個實體的欄位表必須包含：`欄位`、`型別`、`必填`、`說明`、`驗證規則`、`對應 User Story`。
- `必填` 使用 `是`／`否`／`系統產生`；不可改回「可空」語意欄名。
- `驗證規則` 寫規則文字，並以括號附上 FR／US 編號（只列編號、不寫 FR 全文），多個編號用逗號分隔。
- 關鍵實體列出的關鍵屬性，若決定落庫，應能在欄位表或衍生屬性中找到對應；若改以他表／關聯表達，須在簡介或假設交代。

## Good Example

- 這個例子是好的，因為六欄齊全且 FR 只列編號。

```md
| `name` | TEXT | 是 | 相簿可辨識名稱 | 非空白 (FR-001, FR-014) | US-1, US-5 |
```

## Bad Example

- 這個例子是壞的，因為欄位與 FR 寫法都不符合合約。

```md
| name | 可空：否 | FR-001 全文：系統必須允許... |
```

# Rule 5 - 衍生屬性不落庫，並與落庫欄位分開

- Level: `MUST`
- 可由其他表／欄位計算得出、且不應獨立持久化的屬性，必須寫在 `### 衍生屬性`，不可當成落庫欄位。
- 無衍生屬性時，可省略該節或填「無」。
- 範例資料輸出只含落庫欄位（JSON），不要把衍生屬性假裝成表欄位。

## Good Example

- 這個例子是好的，因為計數不落庫且標明計算方式。

```md
### 衍生屬性

- **photo_count**：COUNT(album_photos) ... (FR-007)。純查詢計算，不落地。
```

## Bad Example

- 這個例子是壞的，因為把可計算欄位寫進表內又沒說明來源。

```md
| photo_count | INTEGER | 是 | 照片數 | |
```
