# Rule 1 - 高影響缺口必須先澄清，或在正文標 `[NEEDS CLARIFICATION]`

- Level: `MUST`
- 若缺口會改變實體集合、關聯基數（如 1:N 對 M:N）、主鍵／唯一鍵策略，或會讓多張表同時改寫，優先呼叫 `/clarify`；未拍板前不可寫成看似已確認的實體／DDL。
- 若必須先寫暫定模型供審閱，必須在受影響的實體／約束／相關條列正文內嵌 `[NEEDS CLARIFICATION: …]`，並可在檔末 `## 假設` 註明第一版暫定取捨。
- 標記密度對齊 specify：只標高影響缺口；低風險預設不標標記。

## Good Example

- 這個例子是好的，因為多相簿歸屬未決時正文有標記。

```md
關聯：Photo 與 Album 為 M:N（album_photos）[NEEDS CLARIFICATION: 需確認一張照片可否屬於多個相簿]
```

## Bad Example

- 這個例子是壞的，因為主結構未定卻寫死 DDL 且無標記。

```md
先假設只能屬一本相簿，把三張表都寫完，之後有空再問。
```

# Rule 2 - `[NEEDS CLARIFICATION]` 格式必須與 specify 一致

- Level: `MUST`
- 待澄清標記固定為 `[NEEDS CLARIFICATION: …]`（`NEEDS` 複數），必須明說缺的決策軸。
- 不可使用 `[Need Clarification]`、`[NEED CLARIFICATION]` 等變體作為契約標記。

## Good Example

- 這個例子是好的，因為決策軸清楚。

```md
[NEEDS CLARIFICATION: 需確認一張照片可否屬於多個相簿]
```

## Bad Example

- 這個例子是壞的，因為字串不一致且語意含糊。

```md
[NEED CLARIFICATION: 關聯之後再說]
```

# Rule 3 - 低風險不確定性只寫檔末「假設」，不打待澄清標記

- Level: `MUST`
- 若缺口不致改變 data-plan 主結構，可用第一版假設繼續，寫入檔末 `## 假設`（`- ` 條列）。
- 此類低風險項目**不要**使用 `[NEEDS CLARIFICATION]`。
- 澄清後必須回寫 `## 假設` 與相關實體／約束／DDL。

## Good Example

- 這個例子是好的，因為低風險只進假設。

```md
## 假設

- photo_count 不落庫，由 album_photos 計數
```

## Bad Example

- 這個例子是壞的，因為把計數策略偏好也打成待澄清標記。

```md
[NEEDS CLARIFICATION: photo_count 要不要快取欄]
```

# Rule 4 - 檔案最下方必須有與 spec.md 同格式的「假設」區塊

- Level: `MUST`
- `data-plan.md` 必須在檔案最下方（DDL 之後）提供 `## 假設`，以 `- ` 條列全域前提與本檔第一版設計假設。
- 對已標 `[NEEDS CLARIFICATION]` 且採暫定往前的項目，應在假設中交代暫定取捨。

## Good Example

- 這個例子是好的，因為假設在 DDL 之後。

````md
## DDL

```sql
...
```

## 假設

- 多相簿歸屬正文已標 NEEDS CLARIFICATION；第一版暫定 M:N
````

## Bad Example

- 這個例子是壞的，因為缺少檔末假設區塊。

````md
## DDL

```sql
...
```

（檔案結束）
````

# Rule 5 - 上游 `[NEEDS CLARIFICATION]` 必須被消化，不可默默消失

- Level: `SHOULD`
- 對 `spec.md` 中與持久化模型相關的 `[NEEDS CLARIFICATION]`，應在本輪澄清、在本檔正文保留／改寫對應標記、寫入 `## 假設`，或聲明「本期不影響資料模型／延後」。

## Good Example

- 這個例子是好的，因為上游缺口都有去向。

```md
spec 缺口 A → 已 clarify
spec 缺口 B → 本檔正文保留 [NEEDS CLARIFICATION: …] 並於 ## 假設註暫定
spec 缺口 C → ## 假設 註明不影響本期資料模型
```

## Bad Example

- 這個例子是壞的，因為上游缺口被默默丟掉。

```md
只寫有把握的表，spec 裡的 NEEDS CLARIFICATION 一律不提。
```
