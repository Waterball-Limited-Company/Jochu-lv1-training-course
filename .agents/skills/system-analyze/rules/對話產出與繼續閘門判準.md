# Rule 1 - 硬停與收尾必須使用語意不同的對話段

- Level: `MUST`
- `/system-analyze` 至少輸出兩段對話：**(A) 硬停段**（research＋plan 完成、等「繼續」）與 **(B) 收尾段**（按需介面跑完）。
- 硬停段標題語意必須是「技術研究／系統分析總覽已完成」或等價，**不可**寫成整段系統分析已全部完成。
- 收尾段標題語意必須是「系統分析已完成」或等價，並建議下一步 skill。

## Good Example

- 這個例子是好的，因為兩段標題不會讓使用者誤以為第一次就已全部結束。

```md
硬停：技術研究與系統分析總覽已完成 - …
收尾：系統分析已完成 - …
```

## Bad Example

- 這個例子是壞的，因為硬停就宣告整段完成。

```md
硬停：系統分析已完成 - …（其實還沒跑 data／api／ui）
```

# Rule 2 - 對話段必須含固定區塊；「下一步」採 Speckit 散文式

- Level: `MUST`
- 硬停段與收尾段皆應包含：任務標題行、規格標題、本輪新增產物表（產物／路徑／說明）、重點摘要、下一步。
- 「下一步」必須使用 Speckit 風格：標題 `下一步` 後接**一段散文**（可含 skill 指令路徑），說明目前停在哪、尚未產出什麼、請依序執行什麼；**不可**用可跳過的編號選項清單（例如「1. … 2. 或直接實作」）。
- 硬停段的「下一步」必須依序表達：(1) 請先 Review `technical-research.md` 與 `plan.md`；有問題直接提出、修改或使用 `/clarify`；(2) 確認無誤後回覆「繼續」。硬停段另須列出即將執行的介面清單（可在重點摘要或下一步附近）。未收到「繼續」前不得進入介面委派。
- 收尾段的「下一步」必須固定順序：先 `/e2e-test-plan`，再 `/task-plan`；**禁止**「或直接實作」、跳過 e2e、或把 task-plan 寫成可選。
- 產物表列實際本輪新增／更新的檔案；硬停通常含 `technical-research.md` 與 `plan.md`，收尾再補上本次有跑的介面檔。

## Good Example

- 這個例子是好的，因為硬停先要求 Review，確認後才「繼續」；收尾固定 e2e → task-plan，且為散文式。

```md
（硬停）
下一步

請先 Review `technical-research.md`（技術決策）與 `plan.md`（系統分析總覽）。有問題請直接提出、修改，或使用 `/clarify` 釐清細節；確認無誤後回覆「繼續」，才會依 `plan.md` 按需產出系統介面合約。

（收尾）
下一步

本指令在系統分析完成後結束。請先執行 `/e2e-test-plan` 產出端到端測試計畫，再執行 `/task-plan` 產生實作任務清單。
```

## Bad Example

- 這個例子是壞的，因為硬停未要求 Review、收尾允許跳過。

```md
（硬停）
下一步
1. 若需釐清：先 /clarify
2. 確認無誤請回覆：繼續

（收尾）
建議下一步
1. /e2e-test-plan
2. /task-plan 或直接實作
```

# Rule 3 - 「繼續」不因未關標記而擋下，但必須重列風險

- Level: `MUST`
- 使用者回「繼續」時，即使 `technical-research.md`（及後續介面檔）仍有未關的 `[NEEDS CLARIFICATION: …]` 或 `## 假設` 風險，**仍可進入**按需介面委派。
- 硬停段與收尾段都必須條列當時可見的未關標記與關鍵假設；不可靜默帶過。
- 標記字串必須與 specify 相同：`[NEEDS CLARIFICATION: …]`（注意 `NEEDS` 複數）。

## Good Example

- 這個例子是好的，因為允許繼續但風險可見。

```md
剩餘風險
- [NEEDS CLARIFICATION: 自動成冊粒度日／月／年]
- 假設：第一版暫定以「日」為粒度
→ 使用者回「繼續」後仍委派介面 skill
```

## Bad Example

- 這個例子是壞的，因為有標記就拒絕繼續，與已確認閘門相反。

```md
偵測到 NEEDS CLARIFICATION，禁止「繼續」，必須先清空全部標記。
```
