# Rule 1 - 必須依技術面切分後平行委派 sub-agent

- Level: `MUST`
- 技術研究必須至少依「前端」「後端」「資料」「整合」四個技術面切分，並平行委派 sub-agent 研究；不可只由單一 agent 一次腦補整份決策。
- 各 sub-agent 的輸入至少包含同 package 的 `spec.md` 與本輪已確認之 clarify／假設；輸出應聚焦該技術面的候選方案、取捨理由與風險。
- 若某技術面在本次需求中明顯無工作量（例如無第三方整合），仍應顯式記錄「本面無獨立決策／併入其他決策」，不可默默省略切分意識。

## Good Example

- 這個例子是好的，因為四面都有平行研究，再進入收斂。

```text
前端 agent → Vite + Vanilla、原生 DnD
後端 agent → Express + multer + sharp
資料 agent → SQLite + source_path／縮圖策略
整合 agent → 無外部雲端；本機檔案系統即可
主 agent → 收斂成決策 1～N
```

## Bad Example

- 這個例子是壞的，因為沒有切分就直接寫完整份研究。

```text
我一個人直接把架構到套件全寫完，沒有 sub-agent。
```

# Rule 2 - 主 agent 負責收斂成可 Review 的決策條目

- Level: `MUST`
- 平行研究完成後，必須由主 agent 把各技術面結果收斂成編號決策清單，而不是把四份研究原文並列交差給使用者。
- 收斂時可跨技術面合併（例如「架構」同時涵蓋前後端邊界），但每一則決策必須可獨立 Review，且避免同一選型在多則決策重複定案又互相矛盾。
- 若 sub-agent 之間出現衝突選型，主 agent 必須先消解衝突（必要時 `/clarify`），再寫入 Decision。

## Good Example

- 這個例子是好的，因為四面研究被收成清楚決策，衝突已消解。

```md
## 決策 1: 採用極簡 Web 前後端分離架構
## 決策 2: 前端不引入 runtime framework
## 決策 4: 後端使用最小必要套件...
```

## Bad Example

- 這個例子是壞的，因為直接貼上四份 agent 原文，讀者無法拍板。

```md
### 前端 agent 原文
...
### 後端 agent 原文
...
（前後端對資料庫選型互相矛盾，未處理）
```

# Rule 3 - 技術面研究要對準需求可行性，而非空泛技術百科

- Level: `SHOULD`
- 每個技術面的研究應回答「這個 spec 的功能／約束，要用什麼技術才可行、為什麼、不選什麼」，而不是寫與需求無關的技術總覽。
- 應優先覆蓋會影響後續 data／api／ui-plan 的選型（執行環境、儲存、上傳、預覽、排序互動等）。
- 可略過與本期 FR 無關的進階議題（例如尚未要求的雲端同步協定細節）。

## Good Example

- 這個例子是好的，因為研究直接對上 US／約束。

```text
US3 主頁拖放 → 評估原生 DnD vs SortableJS
US4 平鋪預覽 → 評估是否需要 sharp 縮圖
US6 自動成冊 → 評估 exifr 與缺日期策略
```

## Bad Example

- 這個例子是壞的，因為寫成與需求脫節的百科。

```text
先介紹 2026 年前端框架生態史，再順便提一下相簿。
```
