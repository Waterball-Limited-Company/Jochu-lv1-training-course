# Rule 1 - 資料實體區只寫 API 會用到的欄位形狀與範例，不寫型別系統

- Level: `MUST`
- 每個「資料實體」必須提供 JSON 範例形狀（欄位名 + 範例值）與欄位說明；正式型別、DDL、必填約束真相歸 `data-plan`／DDL 產物。
- 欄位說明標題固定使用「欄位說明（非型別定義）：」，並保留「> 與資料實體 DDL Mapping。」提示。
- 不可在 api-plan 另建完整型別表或伪装成 DDL 定義。

## Good Example

- 這個例子是好的，因為形狀可 Review，型別仍留給 data-plan。

````md
### 實體形狀（欄位 + 範例資料）

> 與資料實體 DDL Mapping。

```json
{ "id": "alb_...", "name": "旅行" }
```

欄位說明（非型別定義）：

| 欄位 | 說明 |
| --- | --- |
| `id` | 相簿識別 |
| `name` | 可辨識名稱 |
````

## Bad Example

- 這個例子是壞的，因為它在 api-plan 重寫了一份型別／DDL。

```md
| 欄位 | 型別 | 必填 | 約束 |
| --- | --- | --- | --- |
| id | TEXT PK | 是 | ULID |
```

# Rule 2 - 實體層必須對齊 User Story

- Level: `MUST`
- 每個資料實體區塊必須有「對應 User Story」表，列出該實體服務的 US 編號與描述。
- 不可出現完全無 US 追溯的實體，也不可把 FR 清單塞進實體層取代 US 對齊。

## Good Example

- 這個例子是好的，因為實體與故事可追溯。

```md
## 資料實體：Album

### 對應 User Story

| US | 描述 |
| --- | --- |
| US-1 | 手動建立相簿並在相簿內加入照片 |
```

## Bad Example

- 這個例子是壞的，因為實體沒有故事對齊。

```md
## 資料實體：Album

（直接開始寫 Endpoint）
```

# Rule 3 - 一份 api-plan 以資料實體切分，實體下掛相關 Endpoint

- Level: `MUST`
- 文件採單一 Markdown；以 `## 資料實體：{Name}` 切分，該實體相關 Endpoint 掛在實體之下。
- 不可改產出 `openapi.yml` 作為本 skill 主產物；可參考 OpenAPI 可讀結構，但輸出契約仍是 Markdown api-plan。
- 跨實體操作應掛在主要資源所屬實體下，並在設計備註或追溯總表交代關係。

## Good Example

- 這個例子是好的，因為切分單位清楚、單檔可 Review。

```md
## 資料實體：Album
### Endpoint：`POST /albums`
## 資料實體：Photo
### Endpoint：`POST /photos/library`
```

## Bad Example

- 這個例子是壞的，因為主產物變成 OpenAPI 檔，偏離本 skill 契約。

```text
specs/001-photo-albums/system-analyze/openapi.yml
```
