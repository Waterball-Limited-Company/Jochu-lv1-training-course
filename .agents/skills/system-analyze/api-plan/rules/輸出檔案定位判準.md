# Rule 1 - 最終 api-plan 必須寫到固定路徑

- Level: `MUST`
- `api-plan` 的最終產物路徑固定為 `specs/<NNN-plan-package>/system-analyze/api-plan.md`。
- 不可把完成稿只留在聊天訊息、暫存筆記或其他任意檔名中，否則後續系統分析與一致性檢查難以穩定接續。
- 若 `system-analyze/` 目錄尚不存在，應先建立該目錄再寫入 `api-plan.md`。

## Good Example

- 這個例子是好的，因為輸出路徑與 skill 契約完全對齊。

```text
specs/001-photo-albums/system-analyze/api-plan.md
```

## Bad Example

- 這個例子是壞的，因為它把 API 合約放到臨時位置，其他流程無法穩定接手。

```text
tmp/api-schema.md
```

# Rule 2 - `plan-package` 必須與上游 spec 目錄對齊

- Level: `MUST`
- `plan-package` 必須沿用同功能的 `specs/<NNN-plan-package>/` 目錄名，採 `NNN-` 三位數前綴加英文小寫 kebab-case slug。
- 若使用者已指定既有 package，應寫入該 package 的 `system-analyze/`，不可另開平行目錄。
- 若尚無 package，應先確認上游 `spec.md` 的 package，再建立對應的 `system-analyze/`。

## Good Example

- 這個例子是好的，因為 api-plan 與 spec 落在同一 package。

```text
specs/003-reading-list/spec.md
specs/003-reading-list/system-analyze/api-plan.md
```

## Bad Example

- 這個例子是壞的，因為 API 合約與 spec 分散在不同 package，追溯會斷開。

```text
specs/003-reading-list/spec.md
specs/004-api-only/system-analyze/api-plan.md
```

# Rule 3 - 標題與小標必須對齊 specify 契約

- Level: `MUST`
- H1 固定為 `# API 計畫：{{FEATURE_NAME}}`，功能中文名與同 package 的 `spec.md` 對齊。
- 小標固定三行、分行書寫：`**功能分支**:`、`**建立日期**:`、`**狀態**:`；其中功能分支值必須以 inline code 包住，且與所在 `plan-package` 相同。
- `**建立日期**:` 使用本次產出日期（`YYYY-MM-DD`）；`**狀態**:` 第一版預設 `草稿`。
- 不可再使用單行 `**功能**｜**日期**｜來源：`，也不可在 H1 塞入實體英文名清單。

## Good Example

- 這個例子是好的，因為標題與小標與 `spec.md` 同構，且功能分支可對回目錄。

```md
# API 計畫：照片相簿整理

**功能分支**: `001-photo-albums`
**建立日期**: 2026-07-20
**狀態**: 草稿
```

## Bad Example

- 這個例子是壞的，因為沿用舊 pipe metadata，且功能欄與目錄名脫節。

```md
# REST API Schema: Album, Photo

**功能**: photo-albums-final-v2 ｜ **日期**: 2026-07-20 ｜ 來源：聊天紀錄
```
