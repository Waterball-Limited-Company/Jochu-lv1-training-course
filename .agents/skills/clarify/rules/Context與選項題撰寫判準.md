# Rule 1 - `Context` 必須足以支撐使用者做判斷

- Level: `MUST`
- 每題都必須先提供一段好懂且足夠的 `Context`，說明這題為什麼重要、它會影響什麼，以及使用者在回答前需要知道的最小背景。
- 若 `Context` 涉及專有名詞、流程節點、技術架構或角色分工，應先用平易近人的方式解釋，再提出問題。
- 不可把 `Context` 寫成空泛開場白，也不可偷塞未經確認的假設當成既定事實。
- 若文字難以說清流程或結構，可直接在對話中附上 Mermaid，幫助使用者做深度判斷。

## Good Example

- 這個例子是好的，因為它先解釋 MVP 與範圍選擇會影響哪些後續決策，使用者能知道自己正在拍板什麼。

```text
你目前已明確的是「要做請假系統」，但還沒有明確第一版要做到多大範圍。
這會直接影響畫面數量、資料表設計、權限模型與開發時程。
這裡的 MVP 指的是第一版最小可用範圍，也就是先把最核心、最不能錯的流程做對。
```

## Bad Example

- 這個例子是壞的，因為它沒有補足任何決策背景，使用者只能猜測這題的真正影響。

```text
我需要你回答這個問題，這很重要。
```

# Rule 2 - `總結之提問` 必須是一句可直接作答的問題

- Level: `MUST`
- `總結之提問` 必須延續前面的 `Context`，濃縮成一句清楚、自然、可直接引出選項的問題。
- 問題本身應讓使用者讀完就知道自己要決定什麼，不可寫成多重追問、模糊追問或口語碎片。
- 若需要多個提問面向，應拆成多題，不可在一個問題句中混入多個決策軸。

## Good Example

- 這個例子是好的，因為使用者能立刻理解要選的是第一版交付範圍。

```text
第一版你希望先交付哪一種範圍？
```

## Bad Example

- 這個例子是壞的，因為它同時混入多件事，而且句子本身不夠明確。

```text
所以你現在是想先做小一點還是說也可能要把通知那些一起弄，或者你其實還沒想好？
```

# Rule 3 - `Options` 必須是可比較的 Markdown 表格

- Level: `MUST`
- 選項區必須使用 Markdown 表格，且每個選項都要有編號。
- 表格至少應包含 `編號`、`選項`、`說明` 三欄，讓使用者能快速比較差異。
- 每個選項的說明應以 1 至 2 句清楚描述該選項的邊界，以及它與其他選項的區隔。
- 必須標示單選或多選，且最後一定要有 `Others`，讓使用者能補充超出既有選項的情況。

## Good Example

- 這個例子是好的，因為格式固定、可比較，而且保留了 `Others` 讓使用者跳出預設答案。

```md
**Options（單選）**

| 編號 | 選項 | 說明 |
| --- | --- | --- |
| 1 | 只做請假申請與主管審核 | 先聚焦最核心流程，不先做 HR 後台與報表。 |
| 2 | 請假申請 + 主管審核 + HR 後台 | 同時納入 HR 的管理需求，但第一版範圍會比較大。 |
| 3 | Others | 我的需求不是以上兩種，想補充真正想做的範圍。 |
```

## Bad Example

- 這個例子是壞的，因為沒有表格、沒有編號，也看不出各選項差異。

```text
A. 小版
B. 大版
C. 其他
```

# Rule 4 - 必須推薦一個選項並說明理由

- Level: `MUST`
- 每題都必須推薦一個最合理的選項，並在選項文字上直接標示 `(推薦：{{原因}})` 形式的簡短理由。
- 推薦理由應指向目前上下文中的效益、風險控制、驗證速度或資訊完整性，不可只寫「我覺得比較好」。
- 若呼叫者 skill 已指定決策偏好或約束，推薦應優先對齊該約束。

## Good Example

- 這個例子是好的，因為推薦理由直接呼應第一版需要先驗證核心流程。

```text
只做請假申請與主管審核（推薦：先鎖定核心流程，最快驗證需求）
```

## Bad Example

- 這個例子是壞的，因為它只有推薦結論，沒有讓使用者知道推薦依據。

```text
只做請假申請與主管審核（推薦）
```

# Rule 5 - clarify 必須直接在對話中提問，不使用 Ask Tool

- Level: `MUST`
- clarify 的提問必須以一般對話訊息直接輸出，讓使用者能在同一段上下文中閱讀 `Context`、Mermaid、問題與選項。
- 不可改用 Ask Tool 或其他表單式互動取代這個輸出格式。
- 若需要保留問答紀錄，也是在使用者回答後整理紀錄，不是把提問本身轉成工具表單。
