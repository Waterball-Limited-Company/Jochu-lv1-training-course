# Rule 1 - 先問影響範圍最大的缺口

- Level: `MUST`
- clarify 應優先盤點會改變需求範圍、核心流程、架構邊界、角色權限、資料模型、驗收標準或不可逆決策的缺口。
- 若呼叫者 skill 已指定提問面向，先以該面向為主；只有在不違反呼叫者要求時，才依一般影響範圍排序補充其他缺口。
- 不應先問只影響命名、微調體驗、枝節格式或可延後補充的低影響細節。

## Good Example

- 這個例子是好的，因為它先確認第一版範圍與審核流程，兩者都會直接影響後續系統設計與工作量。

```text
需求：我要做公司內部請假系統。
先問：
1. 第一版只做請假申請與主管審核，還是也要納入 HR 後台？
2. 請假流程是單層審核、雙層審核，還是依條件切換？
```

## Bad Example

- 這個例子是壞的，因為它先問按鈕文案與配色，卻還沒有確認系統第一版到底要做多大。

```text
需求：我要做公司內部請假系統。
先問：
1. 送出按鈕要寫「提交」還是「送出」？
2. 主色要藍色還是綠色？
```

# Rule 2 - 每題只聚焦一個決策軸

- Level: `MUST`
- 每一題都應只處理單一主要決策軸，讓使用者能清楚比較選項並直接拍板。
- 若兩個疑點會導向不同選項集合，應拆成不同題，不可把多個互相糾纏的決策硬塞進同一題。
- 可在 `Context` 補充前因後果，但 `總結之提問` 只應要求使用者回答一件事。

## Good Example

- 這個例子是好的，因為它把「第一版範圍」與「通知整合方式」拆成兩題，各自都能獨立作答。

```text
題 1：第一版希望先交付哪一種範圍？
題 2：第一版通知與整合最希望先做到哪一種程度？
```

## Bad Example

- 這個例子是壞的，因為它同時問範圍、通知、時程，使用者很難用單一答案作答。

```text
你第一版要不要做 HR 後台、要不要串 Slack，而且希望兩週內上線嗎？
```

# Rule 3 - 單輪最多 3 題，整個 session 最多 5 題

- Level: `MUST`
- 一輪最多輸出 3 題，讓使用者能專注回答，不要把所有不確定性一次傾倒。
- 整個 clarify session 累計最多 5 題；若 5 題內仍無法收斂，應明確說明剩餘風險與卡點，而不是無限制追問。
- 若高影響缺口超過 3 個，先保留最關鍵的 1 至 3 題，其餘待下一輪再問。

## Good Example

- 這個例子是好的，因為它先用 3 題收斂主要決策，等使用者回答後才判斷是否需要第二輪。

```text
Round 1：
1. 第一版範圍
2. 審核流程
3. 通知整合
```

## Bad Example

- 這個例子是壞的，因為它一次列出 8 題，迫使使用者在沒有節奏的情況下處理所有缺口。

```text
Round 1：
1. 第一版範圍
2. 審核流程
3. 通知整合
4. 權限角色
5. 假別規則
6. 報表需求
7. UI 風格
8. 上線時程
```

# Rule 4 - 缺口已足夠清楚時不應為問而問

- Level: `SHOULD`
- 若使用者已明確拍板某個決策，或可由既有上下文直接推出唯一合理解讀，就不應重複提問。
- clarify 的目的是移除高風險不確定性，不是把所有資訊都改寫成問卷。
- 若剩餘問題只屬於實作時可安全採用的低風險預設，應交由呼叫者 skill 帶著明示假設繼續，而不是新增一輪 clarify。
