# Rule 1 - SOP step 必須對應真實動作

- Level: `MUST`
- 每個 step 都必須描述可直接執行的動作，而不是背景說明、前情提要、規則摘要或設計理念。
- 若一段文字只是在解釋「為什麼之後可能要做某事」，但沒有要求當下執行任何動作，就不應獨立成 step。
- 若同一個 step 同時包含真正動作與前情提要，應保留動作，刪除前情提要。

## Good Example

- 這個例子是好的，因為每一步都有明確當下動作，且讀取目標與整理 SOP 都發生在真正需要它們的流程中。

````md
## Phase 1 -- 收斂並完成 SOP

1. READ 讀取目標 skill 內容，確認要新增、修改或重寫的 SOP 範圍。
2. THINK 在整理 phase 與 step 時，按需讀取相關規則，並依實際需要收斂最小可用流程。
3. WRITE 依思考結果撰寫或修改 SOP section。
````

## Bad Example

- 這個例子是壞的，因為第二步與第三步都像前情提要，沒有明確要求當下執行任何動作。

````md
## Phase 1 -- 載入背景

1. READ 讀取目標 skill 內容。
2. THINK 後面可能會需要判斷是否要讀更多規則。
3. READ 這一步是在提醒後續有一些規則很重要。
````

# Rule 2 - 規則檔只在實際讀取的 step 中被點名

- Level: `MUST`
- 只有在某個 step 真的要求讀取某份規則檔時，該 step 才應點名該檔案。
- 後續 `THINK`、`WRITE`、`READ` 檢查步驟應使用「已載入規則」或等價描述，不重複點名已讀過的檔案。
- 若規則檔不是每次都需要，應把它放在按需讀取的 step，而不是在前面預先宣告。

## Good Example

- 這個例子是好的，因為規則檔只在真正讀取時被點名，後續步驟不重複提檔名，也沒有為了載入規則另切一個 phase。

````md
## Phase 1 -- 收斂並完成 SOP

1. READ 讀取目標 skill 內容，確認要新增、修改或重寫的 SOP 範圍。
2. READ 讀取 `rules/sop-格式規範.md`，確認 SOP 的基本格式要求。
3. READ 若需要判斷步驟是否過長、是否混入前情提要，才讀取 `rules/SOP-精簡與按需載入.md`。
4. THINK 依本次已載入規則整理最小可用的 phase 與 step。
5. WRITE 依本次已載入規則撰寫或修改目標 skill 的 SOP section。
6. READ 回頭檢查結果是否符合本次已載入規則；若不符合，立即修正。
````

## Bad Example

- 這個例子是壞的，因為同一份規則檔在後續步驟被反覆點名，模糊了真正的載入時機。

````md
## Phase 1 -- 載入規則與目標

1. READ 讀取 `rules/SOP-精簡與按需載入.md`。

## Phase 2 -- 套用並完成

1. THINK 依 `rules/SOP-精簡與按需載入.md` 判斷是否應該精簡。
2. WRITE 依 `rules/SOP-精簡與按需載入.md` 撰寫內容。
3. READ 檢查結果是否符合 `rules/SOP-精簡與按需載入.md`。
````

# Rule 3 - 預設採最短可執行流程，僅在需要時才載入額外規則

- Level: `SHOULD`
- SOP 應優先保留完成任務所需的最短步驟路徑。
- 若某份規則只在特定判斷情境下才需要，應延後到該判斷發生時再讀取。
- 不應為了說明完整背景而預先加入不一定會執行的 step。
- 當精簡與清楚衝突時，保留必要清楚度，但仍應避免把可延後的載入動作提前。

## Good Example

- 這個例子是好的，因為它和 bad example 處理同一個情境，但只先做所有情況都需要的動作，把額外規則延後到真的需要判斷時才讀取。

````md
## Phase 1 -- 收斂並完成 SOP

1. READ 讀取目標 skill 內容，確認要新增、修改或重寫的 SOP 範圍。
2. READ 讀取 `rules/sop-格式規範.md`，確認 SOP 的基本格式要求。
3. THINK 先依已載入規則整理 phase 與 step，判斷是否還需要額外規則。
4. READ 只有在需要判斷步驟是否過長、是否混入前情提要時，才讀取 `rules/SOP-精簡與按需載入.md`。
5. THINK 依本次已載入規則收斂最小可用的 SOP。
6. WRITE 依思考結果撰寫或修改 SOP section。
7. READ 回頭檢查結果是否符合本次已載入規則；若不符合，立即修正。
````

## Bad Example

- 這個例子是壞的，因為它把不一定需要的規則先全部載入，讓 SOP 變長，也模糊了真正需要判斷的時機。

````md
## Phase 1 -- 預先載入所有規則

1. READ 讀取 `rules/sop-格式規範.md`。
2. READ 讀取 `rules/SOP-精簡與按需載入.md`。
3. READ 讀取所有可能用得到的規則檔。

## Phase 2 -- 再開始思考

1. THINK 依上面所有規則整理 SOP。
````

# Rule 4 - 不應為讀取規則而設立專門 phase

- Level: `MUST`
- Phase 應描述階段成果，而不是單純容納 `READ` 動作。
- 若讀取規則只是為了支援某個判斷或撰寫動作，應把 `READ` 併入該成果 phase，而不是先切出獨立的讀取 phase。
- 不應出現只負責預先載入規則、但尚未開始分析或撰寫的 phase。
- 只有當整理來源材料本身就是獨立階段成果時，才適合用獨立 phase 承載 `READ` 動作。

## Good Example

- 這個例子是好的，因為 phase 是以「完成 SOP 收斂」為成果切分，規則讀取只是其中一部分，不是獨立的 phase 目的。

````md
## Phase 1 -- 收斂並完成 SOP

1. READ 讀取目標 skill 內容，確認要新增、修改或重寫的 SOP 範圍。
2. READ 讀取 `rules/sop-格式規範.md`，確認 SOP 的基本格式要求。
3. THINK 若需要判斷步驟是否過長、是否混入前情提要，再讀取相關規則，並依本次已載入規則收斂 phase 與 step。
4. WRITE 依思考結果撰寫或修改 SOP section。
5. READ 回頭檢查結果是否符合本次已載入規則；若不符合，立即修正。
````

## Bad Example

- 這個例子是壞的，因為第一個 phase 只是為了安置 `READ` 動作，還沒有任何獨立的階段成果。

````md
## Phase 1 -- 載入規則

1. READ 讀取 `rules/sop-格式規範.md`。
2. READ 讀取 `rules/SOP-精簡與按需載入.md`。

## Phase 2 -- 開始思考

1. THINK 依前面讀到的規則整理 SOP。
````
